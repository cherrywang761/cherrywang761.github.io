在A,B,C都完成同步执行后，执行D

前面说过如果有先后顺序，可以使用thread.join()方法，但是如果想让多个线程同步执行，最后在执行D，如何实现？

Tips：普通使用thread.join()会让ABC顺序执行，起不到同步作用。

三个线程同步运行，当三个线程全部完成运行后，D才开始运行

这里我们可以使用CountdownLatch计数器功能

  1. 创建一个定时器，设置一个初始值
  2. countDownLatch.await()进入等待状态，直到计时器变为0
  3. countDownLatch.countDown()在其他线程调用该方法，使计数器减一
  4. 当计数器为0，countDownLatch.await()等待线程中方法将立即退出并执行方法

因此，CountDownLatch适用于一个线程需要等待多个线程的情况。

三个跑步者准备跑步

三个跑步者，在各自准备好了开跑。

这里三个线程A,B,C需要分别进行准备，然后在三个人都准备好后开始同时运行。如何实现？

CountDownLatch用于计数下降，完成计数后的实施者是主线程，具有不可重复性。所以只有一个线程的一个await()会被响应。

CyclicBarrier的下一步动作实施者还是“其他线程”本身，具有往复多次实施动作的特点。我们这里可以使用它来实现。

  1. 首先创建一个公共对象CyclicBarrier，并设置同时等待的线程数
  2. 这些线程会开始同时进行准备，准备好后要等待其他人完成准备工作，所以要调用cyclicBarrier.await()方法来等待其他人
  3. 当所有人都调用cyclicBarrier.await()方法时，意味着都准备好了，那么这些线程将开始同时执行。

序列化与反序列化

发表于 2021-05-18 更新于 2021-05-19 分类于 面试 阅读次数：
本文字数： 1.3k 阅读时长 ≈ 1 分钟

序列化是一种用来处理对象流的机制，是将java对象转换为字节序列的过程

对象流：将对象转换为二进制流byte[]，我们可以对二进制流进行读写操作，流数据也可以在网络中进行传输。

实现方式：

  • 类实现Serializable接口（标记接口，只是一个单纯的标记而已，说明这个类是可以被序列化的）（不想序列化的字段可以用transient修饰）
  • 实现Externalizable接口（是Serializable接口的子类，transient无效）
  • 我们可以使用一个输出流（FileOutputStream构造一个ObjectOutputStream对象）来承载序列化后的对象，并用ObjectOutputStream对象的writeObject(Object obj)方法可以将obj对象写出（存到磁盘中，保存起来了)，想恢复的话可以使用输入流

虽然实现Externalizable接口能带来一定的性能提升，但由于实现ExternaLizable接口导致了编程复杂度的增加，所以大部分时候都是采用实现Serializable接口方式来实现序列化。

什么时候使用序列化？

  1. 对象序列化可以实现分布式对象

应用场景：RMI(远程方法调用)，要利用对象序列化运行远程主机上的服务，就像在本地机上运行对象一样。

RPC的话应该就要序列化，远程调用还可以用json来传数据。。

  1. java对象序列化不仅保留一个对象的数据，而且递归保存对象引用的每个对象的数据

这样子就可以将整个对象层次写入字节流中，可以保存在文件中或在网络连接上传递数据。

此外序列化可以进行对象的“深复制’’,复制对象本身及引用的对象本身。序列化一个对象可能得到整个对象序列。

简而言之序列化作用：

对象、文件、数据有许多不同的格式，很难统一保存和传输，但是都序列化转成字节流后，就都是同样的形式了，可以使用通用的格式保存或传输。想恢复直接反序列化恢复即可， 这样对象还是对象，文件还是文件。

当两个JVM（进程）在进行远程通信时，彼此可以发送各种类型的数据。无论是何种类型的数据，都会以二进制序列的形式在网络上传送。发送方需要把这个Java对象转换为字节序列，才能在网络上传送；接收方则需要把字节序列再恢复为Java对象。

  • 所有涉及到数据传输或者存储的类，严格意义上来说都要加上序列化ID，这也是一种良好的编程习惯。
serialVersionUID

序列化运行时使用一个称为 serialVersionUID 的版本号与每个可序列化类相关联，该序列号在反序列化过程中用于验证序列化对象的发送者和接收者是否为该对象加载了与序列化兼容的类。为它赋予明确的值。显式地定义serialVersionUID有两种用途：

在某些场合，希望类的不同版本对序列化兼容，因此需要确保类的不同版本具有相同的serialVersionUID；

在某些场合，不希望类的不同版本对序列化兼容，因此需要确保类的不同版本具有不同的serialVersionUID。

Tips: 一般人员的密码信息等属性需要标识不被序列化。防止网络传输被窃取，特别是web程序。

大端系统和小端系统

为什么会存在大小端系统？

计算机系统中，每个地址单元对应一个字节（8bit），一种数据类型的数据对象可能占用若干字节。对于这些对象，我们需要建立两个规则：

  • 该对象的地址是什么？
  • 内存中是如何排序这些对象字节的

这也就是常说的字节序和比特序问题。字节序和比特序一般是一致的，要么都是大端，要么都是小端。

什么是大端小端？

大端：低位（字节/比特）放在高地址中，高位（字节/比特）放在低地址中。

数据低位字节内容放在高地址处，高位字节内容放在低地址处。举例 int 类型在32位或者64位的Windows系统中都是占4个字节，也就是 4*8=32 位。储存 int a=1 在大端系统中，如下图所示。

小端：低位（字节/比特）放在低地址中，高位（字节/比特）放在高地址中。

数据低位字节内容放在低地址处，高危字节内容放在高地址处。举例，还是 int a=1在小端系统，如下图所示。

优势，劣势：

小端模式 ：强制转换数据不需要调整字节内容，1、2、4字节的存储方式一样。

大端模式 ：符号位的判定固定为第一个字节，容易判断正负。

一般操作系统都是小端模式；而通讯协议是大端模式；java和平台无关，默认是大端模式

常见的cpu的大小端：
大端：PowerPC、IBM、Sun
小端：x86
ARM既可以工作在大端模式，也可以工作在小端模式

JMM

https://www.jianshu.com/p/8a58d8335270

jmm是什么？

讲述jmm的背景

原来CPU和内存差不多，后来CPU快了，就增加了工作内存（寄存器和高速缓存）多核并发缓存架构

jmm解决的问题？

为了解决硬件和操作系统的差异

jmm的各部分是干嘛的

分为主内存，工作内存和java线程

主内存是公用的，一般主内存就是我们所说的物理内存，然后我们一般是不对主内存直接修改的

如果每次java线程的更改都直接操作主内存，那么性能影响是比较大的。

为了解决这个问题，工作内存就出现了，工作内存就是我们所说的寄存器和高速缓存。

java线程对变量的操作，会在工作内存中进行操作，不能去直接操作主内存

但是这样子我们能感觉到肯定会有线程不安全的问题。

如何解决线程不安全的问题？

回答这个问题前，我们必须先知道

JMM针对于内存，设定了8个指令

  • lock
  • unlock

作用于主内存，是否将一个变量标识为内存独占

  • read
  • load

作用于工作内存，read就是从主内存get数据，load就相当于将主内存数据set到工作内存中

  • use：使用
  • assign：赋值

作用于工作内存，use把工作内存的变量传给执行引擎，assign从执行引擎接收到的值放入到工作内存的变量副本中

  • store
  • write

作用于主内存，store将工作内存的数据传送到主内存中，write就是将传送到主内存中的数据写入到主内存中

JMM对这八个命令指定了很多规则，但是一般都是按照happen-before规则进行执行的。

JMM的模型特征

原子性

JMM保证上面说的8个交互操作指令都是具有原子性的

但是有例外就是，JMM允许虚拟机把没有volatile修饰的64位数据拆分成两个32位数据来进行操作。

其实就是单个指令是具有原子性的，但是多个指令在整体上是没有原子性的。

解决方案：1. 用原子类 2.synchronzied互斥锁。

原子类内部实现是CAS （自旋锁），乐观锁

synchronized内部实现是monitor锁，是悲观锁

可见性

可见性就是说一个线程修改了值，能够让其他线程知道并修改

JMM实现的途径是变量修改后会将新值同步到主内存中，在变量读取前从主内存中刷新变量的值来实现可见性

实现可见性的方案：

  • volatile关键字

volatile关键字的两个作用，一个是保证变量的可见性，一个是防止指令重排序

保证变量可见性：一个变量，被volatile修饰，就意味着线程的本地内存是无效的，当java线程更改该变量，会直接将新值更新到主内存中；其他线程读取该变量时，也会从主内存中直接读取（因为本地内存没有呀）

  • synchronzied

unlock操作之前，必须把变量值同步到主内存中

  • final关键字

final关键字初始化完成，且没有发生this逃逸，其他线程就能看见final修饰的值

有序性

在单线程环境下，程序永远会“有序的”执行，即：线程内表现为串行语义。但是在多线程环境下，程序会出现“乱序”。

这是因为，JMM允许编译器和处理器对指令进行重排序，本意是为了提高效率，但是在多线程并发的情况下，有可能影响并发执行的正确性

解决方案：

  • volatile关键字

volatie的另一个作用，禁止指令重排序，它是通过添加内存屏障的方式来禁止指令重拍序的。

重排序时，不能把后面的指令放到内存屏障前。

  • synchronized

它通过加锁，让一段代码只能由一个线程来执行。

这两个区别就是，synchronized可以修饰一段代码，或者一个方法。但是volatile只能修饰一个变量。

线程通信方式

volatile关键字

该关键字有两个特性，一：可见性；二：禁止指令重排序

其中可见性就是让线程之间进行通信，两个原则保证线程可见性

  • 所有volatile修饰的变量一旦被某个线程更改，必须立即刷新到主内存
  • 所有volatile修饰的变量在使用之前必须重新读取主内存的值
等待通知机制

等待通知机制是通过wait和notify方法来实现的

wait()：当线程调用wait方法，会释放当前的锁，然后让出CPU，进入等待状态

notify()：唤醒沉睡的线程（不会释放锁）

wait要配合synchronized一起使用，如果没有synchronized会报错

既然wait和notify配合synchronized使用能够实现,当然用 ReentrantLock 结合 Condition也可以完成;

join（）

退让方法

当线程A调用另一个线程B的join方法时，当前线程A阻塞等待线程B执行完毕才能继续执行

相当于A退让了，让B先进行

所以join的好处能够保证线程的执行顺序，但是如果调用线程的join方法其实已经失去了并行的意义

虽然存在多个线程，但是本质上还是串行的

最后join的实现其实是基于等待通知机制的。

两种工具类

JUC工具类CountDownLatch

使用一个计数器进行实现。计数器初始值为线程的数量。

当每一个线程完成自己任务后，计数器的值就会减一。

当计数器的值为0时，表示所有的线程都已经完成一些任务，然后在CountDownLatch上等待的线程就可以恢复执行接下来的任务。

CountDownLatch典型用法：1、某一线程在开始运行前等待n个线程执行完毕。将CountDownLatch的计数器初始化为new CountDownLatch(n)，每当一个任务线程执行完毕，就将计数器减1 countdownLatch.countDown()，当计数器的值变为0时，在CountDownLatch上await()的线程就会被唤醒。一个典型应用场景就是启动一个服务时，主线程需要等待多个组件加载完毕，之后再继续执行。

LockSupport

它是一个线程工具类，所有的方法都是静态方法，可以让线程在任意位置阻塞，也可以在任意位置唤醒。

它的内部其实两类主要的方法：park（停车阻塞线程）和unpark（启动唤醒线程）。

（1）wait和notify都是Object中的方法,在调用这两个方法前必须先获得锁对象，但是park不需要获取某个对象的锁就可以锁住线程。

（2）notify只能随机选择一个线程唤醒，无法唤醒指定的线程，unpark却可以唤醒一个指定的线程。

冒泡

原理：相邻两两元素进行比较，不断重复

最佳：O(n)：传的刚好是有序数组，一轮过去啥都没换

最差: O(n^2)

平均：O（n^2）

选排

原理：先选在排，和开头或结尾的数据位置对换

先找出最小(最大)，放到有序数组的开头(结尾)

稳定：都是O(n^2)

插入排序

原理：取头数，作为有序数组第一位，然后后面插入从后往前

最佳O（n）

最差O（n^2）

平均O（n^2）

Shell Sort

原理:改进版的插入排序，引入gap元素

gap = length / 2;

O(log n ^2)

归并排序(稳定)

原理：分组排序的算法，感觉主要就是一种思想（分治法）

你把大数组分成小数组，小数组用的其他算法进行排序，最后再合并成一个大数组

最佳O(N)

最差O（NlogN）

快速排序

原理：（递归）

  1. 从数列中挑一个元素作为“基准”（pivot）

  2. 一般的就是把pivot作为temp，维护两个指针

    然后，前面找小于pivot的数，后面找大于pivot的数，进行交换

3.重复递归

最佳：O（nlogn）

最差：O（n^2）

平均：O（nlogn）

堆排序

原理：

  1. 先把无序数组构建成大堆顶（此时序列的最大值就是堆顶的根节点）
  2. 将根节点和末尾元素交换，则末尾元素成最大值
  3. 将剩余的n-1元素重构成一个新的大堆顶，并重复上面的过程

一般升序用大堆顶，降序用小堆顶
