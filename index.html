钩子线程一般在并发编程中会使用

当JVM进程中没有活跃的非守护进程时（程序执行完了），或者受到系统的中断信号，JVM进程便会退出，而在JVM进程退出前，hook线程就会启动做系统最后的一些辅助性的工作。

hook线程使用场景：

在线上Java程序中经常会遇到进程挂掉，一些状态没有被正确的保存下来，这种情况就需要在JVM关掉时执行一些清理现场的代码。

Java的ShutdownHook就是一种比较好的解决方案

调用场景：

  1. 程序正常退出
  2. 使用System.exit()
  3. 终端使用Ctrl + C触发中断
  4. 系统关闭
  5. 使用Kill pid命令干掉进程

Tips： kill -9 pid命令不会调用JVM注册的钩子，因为Hook线程只有正确接收到退出信号时，才能被正确执行，而kill -9是强制杀死进程。

不要在Hook线程中执行一些耗时的操作，这样会导致程序长时间不能退出

例子1：

当发生了System.exit(int)时，希望在系统退出前，执行一点任务来做一些资源方面的回收操作，ShutdownHook可以达到这个目的。

例子2：如果系统通过Runtime.getRuntime().exec()或new ProcessBuilder() 启动了子进程（Process），该子进程一直在运行，可能就算当前进程退出，子进程也未必退出，但是在业务上需要它子进程结束，就可以使用钩子进程

Q: 钩子函数中的Thread好像都没有start()启动啊？

A: 传入参数是通过new Thread()创建的线程对象，在Java进程调用exit()时，会调用该线程对象的start()方法将其运行起来，所以不要手工先启动了。

原理

源码中有个ApplicationShutdownHooks类，该类用于跟踪和运行在用户层面注的钩子函数

  • 在该类中，通过add方法注册了一个hook，在run()内部运行静态方法runHooks()，顾名思义运行所有注册的钩子函数。源码规定最多可以有10个钩子函数

  • 该类的内部还维护了一个IdentityHashMap，用于存放hook信息，可以通过add()方法来添加，程序中定义的回调线程都放在这里了。

  • 当调用Runnable.getRunnable().addShutdownHook(Thread)时，会间接调用ApplicationShutdownHooks.add(Thread)将线程放到IdentityHashMap中。

  • 当调用System.exit(int)方法时，会间接调用Shutdown.exit(int)方法，再调用sequence()->runHooks()方法

  • 循环中的一个hook对象就是由ApplicationShutdownHooks的匿名块定义的，因此会调用ApplicationShutdownHooks类的run()方法，再调用它的runHooks()方法，runHooks()如图所示：

从该方法中取出所有的thread，然后启动所有线程，最后通过join()方法等待各个线程结束动作。看源码发现每个hook都有一个线程处理，所以是并发的，不是串行执行的

  • Runtime.getRuntime().removeShutdownHook用于删除一个钩子线程。

Hook线程的应用场景&注意事项

防止程序重复执行

Tips：这种防止程序重复执行的策略，也被应用于Mysql服务器，zookeeper，kafka等系统中。

此外，Hook线程也可以执行一些资源释放的操作，如关闭数据库连接，Socket连接等。

具体实现可以在程序启动时，校验是否已经生成 lock 文件，如果已经生成，则退出程序，如果未生成，则生成 lock 文件，程序正常执行，最后再注入 Hook 线程，这样在 JVM 退出的时候，线程中再将 lock 文件删除掉；

为什么MySQL不推荐使用uuid作为主键？ (juejin.cn)

摘要：

自增id做索引和uuid做索引对比

自增id做索引

MySQL中推荐使用自增id（auto_increment)作主键索引

因为自增主键的值是顺序的，所以每一条插入的数据都会在最后一条数据后。

当达到最大填充因子时候（innodb默认的最大填充因子是页大小的15/16，留1/16的空间用作修改使用)

①当满了15/16后，下一条记录就会写入新的页中，一旦数据以auto_increment的方式顺序加载，主键页也会以近乎于顺序的记录填满，提升了页面的最大填充率，不会有页的浪费

②新插入的行一定会在原有的最大数据行的下一行，mysql的定位和寻址速度很快，不会为计算新行的位置做出额外消耗

③减少了页分裂和碎片的产生

使用uuid的索引内部结构

因为uuid相对顺序是无规律的，新的值不一定比之前的值都要大，所以innodb无法做到总是把新行插入到索引的最后，而是需要为新行寻找新的合适的位置从而分配新空间。

该过程需要很多额外的操作：

①写入的目标页很可能已经刷新到磁盘上并且从缓存上移除；亦或者可能还没有加载到缓存中，innodb在插入之前需要先找到该页，并从磁盘读取目标页到内存中，将导致非常多的随机IO

②因为写入是乱序的，innodb不得不频繁的做页分裂操作，以便为新的行分配空间，页分裂导致移动大量数据。

③频繁的页分裂，页会变得稀疏且被不规则填充，最终会出现数据碎片

此外往往将随机值（uuid和雪花id）载入到聚簇索引后，需要做一次optimeize table来重建表并优化页的填充，又要耗费一定时间。

使用自增id的缺点

① 别人能轻易的爬取你的数据库信息，根据你数据库的自增id获取到你的业务增长信息。

②对于高并发情况，innodb在按主键进行插入时会造成明显的锁争用，主键的上界会成为争抢的热点，所有的插入都发生在这里，并发插入会导致间隙锁竞争

③auto_increment锁机制会造成自增锁的抢夺，有一定性能损失。

UUID使用场景

uuid即通用唯一识别码，由32位的16进制数字构成

UUID的作用就是让分布式系统中的所有元素都能有唯一的辨识信息，这样就不需要考虑数据库创建时的名称重复问题。

UUID的缺陷就是在于生成的结果串会比较长。而且无规律，此外上文中也提到了相关的缺陷。

使用场景：

  1. 会话session使用UUID做token
  2. Redis的防误删会使用UUID
  3. 图片视频等多媒体数据不需要有序性情况存储可以使用UUID

数据页的基本样式

  • FIleHeader 文件头：记录页的通用信息，比如上下页的页号，页类型，所有的数据页其实是一个双链表
  • PageHeader 页头：记录本页存储记录的状态信息，比如本页记录数量，槽数量
  • Infimum + supremum 最小与最大记录，是虚拟记录
  • User Records 真正存数据的地方：以链表的形式存储一条条行记录
  • Free Space 存数据空间中尚未使用的区域
  • Page Directory 页目录：页中某些记录的相对位置，用于提升查询效率
  • File Trailer 文件尾：刷盘时校验页是否完整
User Records

一开始其实是没有user records这个空间的。当插入第一条数据的时候，会从free space空间分出出一个空间到user records，当所有空间全部用完就会再去申请一个新的页使用。

user record空间是一个单链表。链表中的各个节点是按照主键值从小到大的顺序进行连接起来的。

Page Directory

如果我们要在一个页中查找指定的一条记录。除了从头遍历user records还有更高效的方法吗?

Page Directory就为这种情况提供了解决方案

innoDB会将一个页中的所有记录划分成若干个组，每个组4-8个记录。

将每个组最后一个记录相对于第一个记录的地址偏移量提取出来存放在页中叫做Page Direcotry的数组中，数组的元素就是这些地址偏移量，也叫做槽

所以在页中根据主键查找记录是很快的

  1. 二分法确定该记录所在的槽，并找到该槽所在分组中主键值最小的那条记录。
  2. 通过next_record属性遍历单链表找到记录。

Tips: 二分法是对Page Directory数组的，而user records单链表只能是逐步遍历

MySQL的设定中，一个表空间内的一组连续的数据页为一个extent（区），默认区的大小为1MB，一个区有64个页，每个数据页大小为16KB（16*64=1024）

一组数据区 = 256个数据区

页合并

参考文章：https://zhuanlan.zhihu.com/p/98818611

在MySQL中，当你删除了一行记录是，实际上记录并没有被物理删除，记录只是被标记（flaged）为删除并且它的空间变成允许其他记录声明访问。

当需要插入数据id正好在这个标记点，就会复用这个空闲空间。

当该页中删除的记录达到MERGE_THRESHOLD(合并的阈值，默认页体积的50%),InnoDB会开始寻找最靠近的页（前后页），如果看看相邻两个数据页利用率是否低于MERGE_THERSHOLD,如果都很小，系统就会将两个页合并以优化空间使用。

合并操作后，前一页保留之前的数据并且获得另一个页的数据，另一个页将变成一个空页，可以容纳新数据。

Tips1: update操作如果让页中数据体积达到类似的阈值点，同样会进行页合并的操作。

即页合并的操作主要看相邻页的利用率如何。

所以常常说MySQL记录的删除和插入都可能会造成空闲空间的增加。

计算机网络体系结构

  • 应用层 ：为特定应用程序提供数据传输服务，例如 HTTP、DNS 等协议。数据单位为报文。
  • 表示层 ：数据压缩、加密以及数据描述，这使得应用程序不必关心在各台主机中数据内部格式不同的问题。
  • 会话层 ：建立及管理会话。
  • 传输层 ：为进程提供通用数据传输服务。由于应用层协议很多，定义通用的传输层协议就可以支持不断增多的应用层协议。运输层包括两种协议：传输控制协议 TCP，提供面向连接、可靠的数据传输服务，数据单位为报文段；用户数据报协议 UDP，提供无连接、尽最大努力的数据传输服务，数据单位为用户数据报。TCP 主要提供完整性服务，UDP 主要提供及时性服务。
  • 网络层 ：为主机提供数据传输服务。而传输层协议是为主机中的进程提供数据传输服务。网络层把传输层传递下来的报文段或者用户数据报封装成分组。
  • 数据链路层 ：网络层针对的还是主机之间的数据传输服务，而主机之间可以有很多链路，链路层协议就是为同一链路的主机提供数据传输服务。数据链路层把网络层传下来的分组封装成帧。
  • 物理层 ：考虑如何传输比特流，物理层的作用是尽可能屏蔽传输媒体和通信手段的差异，使数据链路层感觉不到这些差异。

Q：路由器需要的协议？

A：路由器只有下面三层协议，因为路由器位于网络核心中，不需要为进程或者应用程序提供服务，因此也就不需要传输层和应用层。

物理层（PASS）

链路层

基本三操作：封装成帧，透明传输，差错检测

  • 封装成帧：网络层传下来的分组会添加首部和尾部封装成帧
  • 透明传输：需要对首部和尾部字段进行定界操作，通过转义字符进行判定，用户是察觉不到转义字符的存在的。
  • 差错检测：广泛使用循环冗余检验（CRC）来检测比特差错

常见协议

CSMA/CD协议

CSMA/CD表示载波监听多点接入/碰撞检测

  • 多点接入：主机以多点方式连接到主机上
  • 载波监听：主机需要不停的监听信道
  • 碰撞检测：如果监听到信道已有其他主机正在发送数据，就表示发生碰撞，就要一个碰撞的退避策略。

截断二进制指数退避算法：从离散的整数集合 {0, 1, .., (2k-1)} 中随机取出一个数，记作 r，然后取 r 倍的争用期作为重传等待时间。

PPP协议

PPP协议是用户计算机和ISP进行通信时所使用的协议

MAC地址

MAC地址是链路层地址，长度为6字节（48位），用于唯一标识网卡

有多少网卡就有多少MAC地址。

网络层

网络层可以说是整个互联网的核心，网络层向上只提供简单灵活的、无连接的、经最大努力交互的数据包服务。

使用IP协议，可以把异构的物理网络连接起来，使得在网络层看起来好像是一个统一的网络。

与 IP 协议配套使用的还有三个协议：

  • 地址解析协议 ARP（Address Resolution Protocol）
  • 网际控制报文协议 ICMP（Internet Control Message Protocol）
  • 网际组管理协议 IGMP（Internet Group Management Protocol）

IP分类

由两部分组成：网络号+主机号

CIDR记法：采用在 IP 地址后面加上网络前缀长度的方法，例如 128.14.35.7/20 表示前 20 位为网络前缀。

地址解析协议 ARP

网络层实现主机之间的数据传输，而链路层实现具体每段链路之间的数据传输。因此在通信过程中，IP 数据报的源地址和目的地址始终不变，而 MAC 地址随着链路的改变而改变。

ARP协议就通过IP地址得到MAC地址

ARP内部有个高速缓存记录局域网上的各主机和路由器的 IP 地址到 MAC 地址的映射表。

如果A知道B的IP，但是ARP中没有记录B的MAC映射，那么它会广播通知并获得MAC值写入ARP高速缓存中。

网际控制报文ICMP

ICMP是为了更有效的转发IP数据报和提高交付成功的机会。它被封装在IP数据报中。

ICMP报文分为差错报文和询问报文

常用操作：PING，他就是ICMP的一个重要应用，来测试两台主机之间的连通性

ping原理就是通过向目的主机发送ICMP Echo请求报文

目的主机收到之后会发送Echo回答报文。

传输层

网络中真正通信的并不是主机而是主机中的进程。

传输层提供了进程间的逻辑通信。

传输层最重要的两个协议：UDP和TCP

UDP

用户数据报协议，无连接，是为了保证及时性传输的，没有拥塞控制

面向报文（对应用程序传下来的报文不合并也不拆分，只添加UDP首部）

TCP

传输控制协议TCP，面向连接，是为了保证可靠性传输的，有流量控制，拥塞控制，面向字节流（应用层传下来的报文当做字节流，把字节流组织成大小不等的数据块），TCP只能一对一

三握四挥

三次握手

Client想和Server建立连接，于是主动发送TCP握手请求

  • C向S发送连接诶请求报文，SYN = 1, ACK = 0, 选择一个初始序号seq= x
  • S接收到请求报文，如果同意建立连接，向A发送连接确认报文，SYN = 1, ACK = 1, 确认号ack = x+1, 同时也选择一个初始序号 seq= y;
  • C收到S发送的确认报文后，还要最终核实一下，向B发送确认, ACK = 1, 确认号ack = y+1，序号为x+1
  • S收到确认后，建立连接

为什么要三次握手？

保证C，S的收发都没有问题，这样能防止失效的连接请求到服务器，让服务器错误打开连接，消耗资源

第三次握手取消掉行不行？

不行啊

  1. 如果取消掉的话，S端不知道C收和S发正不正常

  2. 倘若有网络问题，S端第二次握手返回的确认报文很慢，C端等了一个超时重传时间还没到，它就要重新请求连接。

    结果重新请求完，这个确认报文到了，如果没有第三次握手，那么服务器就会对滞留的连接请求进行确认（但实际上它应该是过期了）

四次挥手

在建立连接后ACK = 1

Client想与Server结束连接

  • A发送连接释放报文， FIN = 1,选择初始序号seq = u
  • B收到释放报文后进行确定，ACK=1,ack = u+1, 选择一个初始序号seq=v，这时候TCP属于半关闭状态（CLOSE WAIT)，B能向A发送数据但是A不能向B发送数据
  • 等B发送完数据后，不需要维持这个连接了，就发送释放报文给A，FIN=1,ACK=1,ack=u+1,选择一个初始序号seq = w
  • A收到后，进入TIME-WAIT状态，等待2MSL(最大报文存活时间)后释放连接，ACK=1, ack = w+1, seq = u+1
  • B收到A的确认后释放连接

为什么要四次挥手？

  1. CLOSE_WAIT过程是为了让服务端发送完所有数据。
  2. TIME_WAIT过程的2MSL时间：（为什么要等2MSL呢？）
  1. 确保最后一个A发向B的确认报文能够到达。如果B没收到A发送的确认报文，那么会重新发送连接释放请求报文，等2MSL这段时间就是为了处理这种情况
  2. 等一段时间也是为了让本次连接产生的所有报文都从网络中消失，使得下次新连接出现旧连接报文

拥塞控制

拥塞控制为了降低整个网络的拥塞程度。

因为如果网络出现拥塞，分组将会丢失。所以发送方会再次重传数据，从而导致网络的拥塞程度更高，这样的恶性循环必须避免，因此当出现拥塞时，应当控制发送方的速率。

TCP主要通过四个算法来进行拥塞控制：

慢开始、拥塞避免、快重传、快恢复

cwnd：拥塞窗口，是一个状态变量，但实际发送多少数据取决于发送方窗口

ssthresh：慢开始门限

  • 慢开始

最初令cwnd为1，当cwnd < ssthresh时，每次cwnd加倍

当cwnd >= ssthresh时，进入拥塞避免

  • 拥塞避免

当进入拥塞避免：每个轮次cwnd加1

如果超时，令ssthresh = cwnd / 2，并重新执行慢开始

  • 快重传

发送方传输数据给接收方，先发送M1数据，接收方接收到并返回接收M1数据的确认报文，接着发送M2，假设发送M3数据时，数据丢失了，当发送方发送M4数据时，接收方发现M3没有确认，就会要求重复确认M2，发送M5，再次要求重复确认M2，发送M6，第三次重复确认M2，因为三次重复确认请求就会触发快重传机制，这时候发送方的第一要务就是把M3给发出来..

  • 快恢复

分布式的意思就是每个站点只保留它自己的那部分数据

  • 区域传送（区域传送指的是主域名服务器向辅助域名服务器传送变化的那部分数据）

文件传输协议（FTP）

A: 如果有两台不同的主机，使用不同的操作系统，不同的字符集，如何进行文件传输呢？

Q: 可以采用FTP的传输方式，也可以使用Telnet协议进行不同操作系统之间的文件传输。但是FTP与Telnet的区别在于FTP使用2个TCP连接传输文件，且不需要强制两端都使用同一个标准。

为什么要使用两个连接呢？

要是用一条连接的话，端口既要处理数据连接数据，又要处理控制连接，很容易出错，而且效率很低。区分控制连接和数据连接后，数据连接的速度要快一些。

2个连接
  • 控制连接

控制连接使用端口21，主要用于在两个主机间传输控制信息，如用户标识、口令相关数据，相关命令从客户端传送给服务端，同时将服务端的应答传送给客户端

  • 数据连接

用于传输一个文件数据，使用端口20

2种模式

根据数据连接是否是让服务端主动建立的，分为主动模式和被动模式

而两者建立连接的过程，就是两次TCP请求的构建，只不过两种模式的数据连接构建并不大相同。。

  • 主动模式

服务端主动建立数据连接，服务器端的端口号为20，客户端端口号随机，但必须大于1024（因为0~1023是熟知端口号）

主动模式的利弊

主动模式对FTP服务器的管理有利，因为FTP服务器只需要开启21端口的“准入”和20端口的“准出”即可。

但这种模式对客户端的管理不利，因为FTP服务器20端口连接客户端的数据端口时，有可能被客户端的防火墙拦截掉。

  • 被动模式

服务端不是主动建立数据连接，而是由客户端主动建立数据，且客户端的端口号也由自己决定，服务端端口号随机

被动模式的利弊

被动模式对FTP客户端的管理有利，因为客户端的命令端口和数据端口都是“准出”，windows防火墙对于“准出”一般是不拦截的，所以客户端不需要任何多余的配置就可以连接FTP服务器了。

但对服务器端的管理不利。因为客户端数据端口连到FTP服务器的数据端口P时，很有可能被服务器端的防火墙阻塞掉。

动态主机配置协议（DHCP）

相关信息包括：IP地址，子网掩码，网关IP地址等

  • 最后通过DHCP报文，配置IP地址，子网掩码和DNS服务器的IP地址

  • 主机生成ARP查询报文，然后封装报文成帧，通过交换机转发该帧给所有连接设备
  • 路由器收到该帧后，向上分解得到ARP报文，进行匹配，成功就返回一个ARP回答报文给主机，其中包括它的MAC地址。

  1. DNS解析域名

  • 知道网关路由器的MAC地址后，就可以进行DNS的解析过程了。

  • 三次握手建立连接后，浏览器生成HTTP GET报文，获得HTTP响应报文，抽取出存放在报文主体的Web页面内容，进行渲染显示出Web页面

计算机网络体系结构

  • 应用层 ：为特定应用程序提供数据传输服务，例如 HTTP、DNS 等协议。数据单位为报文。
  • 表示层 ：数据压缩、加密以及数据描述，这使得应用程序不必关心在各台主机中数据内部格式不同的问题。
  • 会话层 ：建立及管理会话。
  • 传输层 ：为进程提供通用数据传输服务。由于应用层协议很多，定义通用的传输层协议就可以支持不断增多的应用层协议。运输层包括两种协议：传输控制协议 TCP，提供面向连接、可靠的数据传输服务，数据单位为报文段；用户数据报协议 UDP，提供无连接、尽最大努力的数据传输服务，数据单位为用户数据报。TCP 主要提供完整性服务，UDP 主要提供及时性服务。
  • 网络层 ：为主机提供数据传输服务。而传输层协议是为主机中的进程提供数据传输服务。网络层把传输层传递下来的报文段或者用户数据报封装成分组。
  • 数据链路层 ：网络层针对的还是主机之间的数据传输服务，而主机之间可以有很多链路，链路层协议就是为同一链路的主机提供数据传输服务。数据链路层把网络层传下来的分组封装成帧。
  • 物理层 ：考虑如何传输比特流，物理层的作用是尽可能屏蔽传输媒体和通信手段的差异，使数据链路层感觉不到这些差异。

Q：路由器需要的协议？

A：路由器只有下面三层协议，因为路由器位于网络核心中，不需要为进程或者应用程序提供服务，因此也就不需要传输层和应用层。

链路层

基本三操作：封装成帧，透明传输，差错检测

  • 封装成帧：网络层传下来的分组会添加首部和尾部封装成帧
  • 透明传输：需要对首部和尾部字段进行定界操作，通过转义字符进行判定，用户是察觉不到转义字符的存在的。
  • 差错检测：广泛使用循环冗余检验（CRC）来检测比特差错

使用IP协议，可以把异构的物理网络连接起来，使得在网络层看起来好像是一个统一的网络。

地址解析协议 ARP

网络层实现主机之间的数据传输，而链路层实现具体每段链路之间的数据传输。因此在通信过程中，IP 数据报的源地址和目的地址始终不变，而 MAC 地址随着链路的改变而改变。

ARP协议就通过IP地址得到MAC地址

ARP内部有个高速缓存记录局域网上的各主机和路由器的 IP 地址到 MAC 地址的映射表。

如果A知道B的IP，但是ARP中没有记录B的MAC映射，那么它会广播通知并获得MAC值写入ARP高速缓存中。

传输层

网络中真正通信的并不是主机而是主机中的进程。

传输层提供了进程间的逻辑通信。

传输层最重要的两个协议：UDP和TCP

UDP

用户数据报协议，无连接，是为了保证及时性传输的，没有拥塞控制

面向报文（对应用程序传下来的报文不合并也不拆分，只添加UDP首部）

TCP

传输控制协议TCP，面向连接，是为了保证可靠性传输的，有流量控制，拥塞控制，面向字节流（应用层传下来的报文当做字节流，把字节流组织成大小不等的数据块），TCP只能一对一

三握四挥

三次握手

Client想和Server建立连接，于是主动发送TCP握手请求

  • C向S发送连接诶请求报文，SYN = 1, ACK = 0, 选择一个初始序号seq= x
  • S接收到请求报文，如果同意建立连接，向A发送连接确认报文，SYN = 1, ACK = 1, 确认号ack = x+1, 同时也选择一个初始序号 seq= y;
  • C收到S发送的确认报文后，还要最终核实一下，向B发送确认, ACK = 1, 确认号ack = y+1，序号为x+1
  • S收到确认后，建立连接

为什么要三次握手？

保证C，S的收发都没有问题，这样能防止失效的连接请求到服务器，让服务器错误打开连接，消耗资源

第三次握手取消掉行不行？

不行啊

  1. 如果取消掉的话，S端不知道C收和S发正不正常

  2. 倘若有网络问题，S端第二次握手返回的确认报文很慢，C端等了一个超时重传时间还没到，它就要重新请求连接。

    结果重新请求完，这个确认报文到了，如果没有第三次握手，那么服务器就会对滞留的连接请求进行确认（但实际上它应该是过期了）

四次挥手

在建立连接后ACK = 1

Client想与Server结束连接

  • A发送连接释放报文， FIN = 1,选择初始序号seq = u
  • B收到释放报文后进行确定，ACK=1,ack = u+1, 选择一个初始序号seq=v，这时候TCP属于半关闭状态（CLOSE WAIT)，B能向A发送数据但是A不能向B发送数据
  • 等B发送完数据后，不需要维持这个连接了，就发送释放报文给A，FIN=1,ACK=1,ack=u+1,选择一个初始序号seq = w
  • A收到后，进入TIME-WAIT状态，等待2MSL(最大报文存活时间)后释放连接，ACK=1, ack = w+1, seq = u+1
  • B收到A的确认后释放连接

为什么要四次挥手？

  1. CLOSE_WAIT过程是为了让服务端发送完所有数据。
  2. TIME_WAIT过程的2MSL时间：（为什么要等2MSL呢？）
  1. 确保最后一个A发向B的确认报文能够到达。如果B没收到A发送的确认报文，那么会重新发送连接释放请求报文，等2MSL这段时间就是为了处理这种情况
  2. 等一段时间也是为了让本次连接产生的所有报文都从网络中消失，使得下次新连接出现旧连接报文

拥塞控制

拥塞控制为了降低整个网络的拥塞程度。

因为如果网络出现拥塞，分组将会丢失。所以发送方会再次重传数据，从而导致网络的拥塞程度更高，这样的恶性循环必须避免，因此当出现拥塞时，应当控制发送方的速率。

TCP主要通过四个算法来进行拥塞控制：

慢开始、拥塞避免、快重传、快恢复

cwnd：拥塞窗口，是一个状态变量，但实际发送多少数据取决于发送方窗口

ssthresh：慢开始门限

  • 慢开始

最初令cwnd为1，当cwnd < ssthresh时，每次cwnd加倍

当cwnd >= ssthresh时，进入拥塞避免

  • 拥塞避免

当进入拥塞避免：每个轮次cwnd加1

如果超时，令ssthresh = cwnd / 2，并重新执行慢开始

  • 快重传

发送方传输数据给接收方，先发送M1数据，接收方接收到并返回接收M1数据的确认报文，接着发送M2，假设发送M3数据时，数据丢失了，当发送方发送M4数据时，接收方发现M3没有确认，就会要求重复确认M2，发送M5，再次要求重复确认M2，发送M6，第三次重复确认M2，因为三次重复确认请求就会触发快重传机制，这时候发送方的第一要务就是把M3给发出来..

  • 快恢复

文件传输协议（FTP）

A: 如果有两台不同的主机，使用不同的操作系统，不同的字符集，如何进行文件传输呢？

Q: 可以采用FTP的传输方式，也可以使用Telnet协议进行不同操作系统之间的文件传输。但是FTP与Telnet的区别在于FTP使用2个TCP连接传输文件，且不需要强制两端都使用同一个标准。

为什么要使用两个连接呢？

要是用一条连接的话，端口既要处理数据连接数据，又要处理控制连接，很容易出错，而且效率很低。区分控制连接和数据连接后，数据连接的速度要快一些。

2个连接
  • 控制连接

控制连接使用端口21，主要用于在两个主机间传输控制信息，如用户标识、口令相关数据，相关命令从客户端传送给服务端，同时将服务端的应答传送给客户端

  • 数据连接

用于传输一个文件数据，使用端口20

2种模式

根据数据连接是否是让服务端主动建立的，分为主动模式和被动模式

而两者建立连接的过程，就是两次TCP请求的构建，只不过两种模式的数据连接构建并不大相同。。

  • 主动模式

服务端主动建立数据连接，服务器端的端口号为20，客户端端口号随机，但必须大于1024（因为0~1023是熟知端口号）

主动模式的利弊

主动模式对FTP服务器的管理有利，因为FTP服务器只需要开启21端口的“准入”和20端口的“准出”即可。

但这种模式对客户端的管理不利，因为FTP服务器20端口连接客户端的数据端口时，有可能被客户端的防火墙拦截掉。

  • 被动模式

服务端不是主动建立数据连接，而是由客户端主动建立数据，且客户端的端口号也由自己决定，服务端端口号随机

被动模式的利弊

被动模式对FTP客户端的管理有利，因为客户端的命令端口和数据端口都是“准出”，windows防火墙对于“准出”一般是不拦截的，所以客户端不需要任何多余的配置就可以连接FTP服务器了。

但对服务器端的管理不利。因为客户端数据端口连到FTP服务器的数据端口P时，很有可能被服务器端的防火墙阻塞掉。

  • 主机生成DHCP请求报文，将该报文放入具有目的端口67和源端口68的UDP报文段中

  • IP数据报需要放置在MAC帧中，将广播到与交换机连接的所有设备上
  • 连接在交换机的DHCP服务器收到广播帧后，会不断向上分解得到IP数据报，UDP数据报、DHCP请求报文，根据DHCP请求报文会得到DHCP ACK报文。。。在封装（UDP握手）
  • 最后通过DHCP报文，配置IP地址，子网掩码和DNS服务器的IP地址

  1. ARP解析MAC地址

因为DHCP过程只能知道网关路由器的IP地址，为了获取网关路由器的MAC地址，需要使用ARP协议

  • 主机通过浏览器生成TCP套接字，该套接字向HTTP服务器发送HTTP请求。为了生成套接字，需要知道网站域名

  • 路由器收到该帧后，向上分解得到ARP报文，进行匹配，成功就返回一个ARP回答报文给主机，其中包括它的MAC地址。

  • 知道网关路由器的MAC地址后，就可以进行DNS的解析过程了。

  • 三次握手建立连接后，浏览器生成HTTP GET报文，获得HTTP响应报文，抽取出存放在报文主体的Web页面内容，进行渲染显示出Web页面

前面说过如果有先后顺序，可以使用thread.join()方法，但是如果想让多个线程同步执行，最后在执行D，如何实现？

Tips：普通使用thread.join()会让ABC顺序执行，起不到同步作用。

三个线程同步运行，当三个线程全部完成运行后，D才开始运行

这里我们可以使用CountdownLatch计数器功能

  1. 创建一个定时器，设置一个初始值
  2. countDownLatch.await()进入等待状态，直到计时器变为0
  3. countDownLatch.countDown()在其他线程调用该方法，使计数器减一
  4. 当计数器为0，countDownLatch.await()等待线程中方法将立即退出并执行方法

三个跑步者准备跑步

三个跑步者，在各自准备好了开跑。

这里三个线程A,B,C需要分别进行准备，然后在三个人都准备好后开始同时运行。如何实现？

CountDownLatch用于计数下降，完成计数后的实施者是主线程，具有不可重复性。所以只有一个线程的一个await()会被响应。

CyclicBarrier的下一步动作实施者还是“其他线程”本身，具有往复多次实施动作的特点。我们这里可以使用它来实现。

  1. 首先创建一个公共对象CyclicBarrier，并设置同时等待的线程数

  3. 当所有人都调用cyclicBarrier.await()方法时，意味着都准备好了，那么这些线程将开始同时执行。

子线程将结果返回到主线程

在开发中，我们会遇到需要fork子进程执行一些耗时任务，执行完后将结果返回给主线程，那么如何具体实现呢?

需要返回结果，显然需要使用Callable接口来实现

Callable通常会和FutureTask结合使用

序列化与反序列化

发表于 2021-05-18 更新于 2021-05-19 分类于 面试 阅读次数：

序列化是一种用来处理对象流的机制，是将java对象转换为字节序列的过程

对象流：将对象转换为二进制流byte[]，我们可以对二进制流进行读写操作，流数据也可以在网络中进行传输。

实现方式：

  • 类实现Serializable接口（标记接口，只是一个单纯的标记而已，说明这个类是可以被序列化的）（不想序列化的字段可以用transient修饰）
  • 实现Externalizable接口（是Serializable接口的子类，transient无效）
  • 我们可以使用一个输出流（FileOutputStream构造一个ObjectOutputStream对象）来承载序列化后的对象，并用ObjectOutputStream对象的writeObject(Object obj)方法可以将obj对象写出（存到磁盘中，保存起来了)，想恢复的话可以使用输入流

虽然实现Externalizable接口能带来一定的性能提升，但由于实现ExternaLizable接口导致了编程复杂度的增加，所以大部分时候都是采用实现Serializable接口方式来实现序列化。

什么时候使用序列化？

  1. 对象序列化可以实现分布式对象

应用场景：RMI(远程方法调用)，要利用对象序列化运行远程主机上的服务，就像在本地机上运行对象一样。

RPC的话应该就要序列化，远程调用还可以用json来传数据。。

  1. java对象序列化不仅保留一个对象的数据，而且递归保存对象引用的每个对象的数据

这样子就可以将整个对象层次写入字节流中，可以保存在文件中或在网络连接上传递数据。

此外序列化可以进行对象的“深复制’’,复制对象本身及引用的对象本身。序列化一个对象可能得到整个对象序列。

简而言之序列化作用：

对象、文件、数据有许多不同的格式，很难统一保存和传输，但是都序列化转成字节流后，就都是同样的形式了，可以使用通用的格式保存或传输。想恢复直接反序列化恢复即可， 这样对象还是对象，文件还是文件。

当两个JVM（进程）在进行远程通信时，彼此可以发送各种类型的数据。无论是何种类型的数据，都会以二进制序列的形式在网络上传送。发送方需要把这个Java对象转换为字节序列，才能在网络上传送；接收方则需要把字节序列再恢复为Java对象。

  • 所有涉及到数据传输或者存储的类，严格意义上来说都要加上序列化ID，这也是一种良好的编程习惯。
serialVersionUID

序列化运行时使用一个称为 serialVersionUID 的版本号与每个可序列化类相关联，该序列号在反序列化过程中用于验证序列化对象的发送者和接收者是否为该对象加载了与序列化兼容的类。为它赋予明确的值。显式地定义serialVersionUID有两种用途：

在某些场合，希望类的不同版本对序列化兼容，因此需要确保类的不同版本具有相同的serialVersionUID；

在某些场合，不希望类的不同版本对序列化兼容，因此需要确保类的不同版本具有不同的serialVersionUID。

Tips: 一般人员的密码信息等属性需要标识不被序列化。防止网络传输被窃取，特别是web程序。

大端系统和小端系统

为什么会存在大小端系统？

计算机系统中，每个地址单元对应一个字节（8bit），一种数据类型的数据对象可能占用若干字节。对于这些对象，我们需要建立两个规则：

  • 该对象的地址是什么？
  • 内存中是如何排序这些对象字节的

这也就是常说的字节序和比特序问题。字节序和比特序一般是一致的，要么都是大端，要么都是小端。

什么是大端小端？

大端：低位（字节/比特）放在高地址中，高位（字节/比特）放在低地址中。

数据低位字节内容放在高地址处，高位字节内容放在低地址处。举例 int 类型在32位或者64位的Windows系统中都是占4个字节，也就是 4*8=32 位。储存 int a=1 在大端系统中，如下图所示。

小端：低位（字节/比特）放在低地址中，高位（字节/比特）放在高地址中。

数据低位字节内容放在低地址处，高危字节内容放在高地址处。举例，还是 int a=1在小端系统，如下图所示。

优势，劣势：

小端模式 ：强制转换数据不需要调整字节内容，1、2、4字节的存储方式一样。

大端模式 ：符号位的判定固定为第一个字节，容易判断正负。

一般操作系统都是小端模式；而通讯协议是大端模式；java和平台无关，默认是大端模式

常见的cpu的大小端：
大端：PowerPC、IBM、Sun
小端：x86
ARM既可以工作在大端模式，也可以工作在小端模式
