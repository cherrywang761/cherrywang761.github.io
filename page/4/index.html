进程同步机制

进程同步就是说不同的进程之间传递信息。它是一组编程接口，让程序员来协调进程，使之能够相互通信

进程通信（IPC）

键相当于IPC的外部名，标识符相当于IPC的内部名，IPC=内核对象（进程通信是靠内核对象沟通的）

IPC的目的

网络socket

一般在TCP/IP通信中比较常见，是OS提供给程序用来与TCP/IP协议交互的接口，socket套接字，两个程序进行网络通信时，通信连接的端点就可以理解为两个socket

socket表现形式 IP：端口

程序交互=进程交互，程序的执行过程相当于进程从创建到运行到消亡的这么一个过程

网络的进程通信中是十分常见的

socket适用于同一主机不同进程间的通信（模块之间feign调用）和不同主机进程间的全双工网络通信。。

管道

管道分为三种，传输数据都是无类型的，根据FIFO机制

  • 匿名管道：也叫普通管道：PIPE：两个限制：1.只能父子进程，兄弟进程通信 2. 半双工通信，数据只能单向传输（对讲机）
  • 命令流管道：进行了优化，它是全双工通信（电话），但是第一个限制没有解除
  • 命名管道：FIFO，两个限制都解除了

一般就是喜欢拿匿名管道和命名管道做比较

  1. 匿名管道是在内存中的（fork），但是命名管道是在文件系统中的，它是关联了一个路径，以FIFO文件形式存储在文件系统中的，设备文件，所以它打开要用open()才能打开，而且可能阻塞
  2. 匿名管道用pipe函数创建打开，命名函数用mkfifo函数创建，open打开

管道也是文件嘛，大小4096字节，管道满则写阻塞，管道空则读阻塞

系统IPC

消息队列

消息队列遵循FIFO原则

一个系统最多3636个消息队列

每个消息队列最多可以装8196字节

一个消息队列所有消息最多可以存放16384个字节

一般应用上我们用消息队列发送数据，用信号量进行进程的同步控制

和管道区别：1. 数据有无类型，管道无类型，所以FIFO机制，消息队列有类型，所以可以设优先级机制

  1. 读数据的顺序，管道先来先读，消息队列有优先级机制

信号量

临界资源：同一时刻，只能被同一进程访问的共享资源

临界区：访问临界资源的代码

信号量其实就是一个计数器：记录临界资源能够被多少进程同时访问

一般是用在进程通信方面，或者说同一进程不同线程的同步方法

共享内存

shmget（）开辟共享内存

shmdt() 断开进程和共享内存的链接

shmctl() 删除共享内存

没有连接就会删除，删除后就不会有连接（连接不上）

共享内存是最快的IPC方式，它其实就是映射一段能被其他进程访问的内存。

这个内存是由一个进程创建的，但是其他进程也可以访问。

之所以快是因为不需要客户进程和服务器进程之间进行赋值

共享内存唯一要特别注意的就是同步问题了，所以在这里使用信号量进行共享内存的进程同步是非常常见的。

共享内存只需要两次拷贝，而比如管道，消息队列这些内核对象执行系统调用，都需要4次拷贝

消息队列4次拷贝：

  1. 用户buffer -> kernel
  2. kernel -> 内存
  3. 内存 -> kernel
  4. kernel -> 用户buffer

共享内存2次：

  1. 用户空间 -> 内存
  2. 内存 -> 用户空间

RPC框架

系统架构从以前的单一系统走向分布式系统，好处：

  • 对业务模块的拆分，能够使得各个模块的职责更加清晰明了（虽然模块职责边界划分十分头疼。。）
  • 对业务模块细致的划分能使得项目在管理上面会更加方便，从代码角度上看，开发和维护的成本也会降低。
  • 提高了系统的容错率，单一系统如果宕机就GG，分布式一般都会使用冗余的方式来提高可用性（坏处就是成本提高了。。）

而随着集中式架构向分布式架构进行转变，应用系统之间的服务调用于通讯问题成为了首要解决的需求。

所以RPC站了出来，它就是用来让分布式应用更加简单的。它让调本地方法和调其他模块的方法在外表上看起来是一样的。。（RPC 框架需提供一种透明调用机制让使用者不必显式的区分本地调用和远程调用。）

RPC 的全称是 Remote Procedure Call 是一种进程间通信方式。 它允许程序调用另一个进程上（通常是共享网络的另一台机器上）的过程或函数，而不用程序员显式编码这个远程调用的细节。

RPC解决三件事（公平，公平，还是tm的公平）

  • 进程间通讯
  • 提供和本地方法调用一样的调用机制
  • 屏蔽程序员对远程调用的细节实现

进程通信问题

​ 对于分布式环境，RPC能够帮助我们解决不同服务器之间的通信及数据传输问题（能够做好方法调用到数据的转换，然后借助网络进行数据传递的内部转换）

​ rpc客户端向rpc服务端发起远程服务调用。

通过请求的封装，参数的封装，序列化，编码，约定协议传输，解析请求，处理请求，封装返回消息数据，再进行返回数据的序列化， 编码，再通过网络返回给客户端

上述的操作，都需要进行封装，屏蔽远程调用的细节

什么样的内容封装才能让我们觉得像以本地调用方式调用远程服务呢？

显而易见：代理

代理有两种方式

  • jdk动态代理（接口代理）
  • cglib代理（子类代理）

RPC还是倾向于动态代理，尽管字节码方式高效强大，但是动态代理易于维护

基本原理

rpc需要对远程调用内部细节封装，主要由以下几个方面

  • 序列码
  • 编解码
  • 协议
  • 网络

从发起远程调用——>接受数据返回结果，过程如下：

  1. client以本地调用方式调用服务
  2. client stub（客户端存根）接受调用负责将方法参数组装成能够进行网络传输的消息体
  3. client stub 找到服务地址，将消息发送给服务端
  4. server stub（服务端存根）接收到消息后进行解码
  5. server stub根据解码结果调用本地服务
  6. 本地服务执行将结果返回给server stub
  7. server stub将返回结果打包成消息体发送给消费方
  8. client stub接受信息并解码
  9. client stub将结果传输给client，消费方得到最终结果

RPC框架相当于封装了step2-step8这些步骤，对外只暴露client和client stub沟通的接口

进一步的拆解

基本的形式要记住，记住方便理解

​ RPC 服务端通过 RpcServer 去暴露服务接口，而客户端通过 RpcClient 去获取服务接口。

​ 客户端像调用本地方法一样去调用远程接口方法，RPC 框架提供接口的代理实现，实际的调用将委托给代理 RpcProxy。

​ 代理封装调用信息并将调用转交给 RpcInvoker 去实际执行。

​ 在客户端的 RpcInvoker 通过连接器 RpcConnector 去维持与服务端的通道 RpcChannel，并使用 RpcProtocol 执行协议编码（encode）并将编码后的请求消息通过通道发送给服务端。

​ RPC 服务端接收器 RpcAcceptor 接收客户端的调用请求，同样使用 RpcProtocol 执行协议解码（decode）。

​ 解码后的调用信息传递给 RpcProcessor 去控制处理调用过程，最后再委托调用给 RpcInvoker 去实际执行并返回调用结果。

  • RpcServer（生产者接口）
  • RpcClient（消费者接口）
  • RpcProxy（代理服务）
  • RpcProtocol（编解码服务）
  • 网络连接器
