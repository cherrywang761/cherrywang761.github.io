冒泡

原理：相邻两两元素进行比较，不断重复

最佳：O(n)：传的刚好是有序数组，一轮过去啥都没换

最差: O(n^2)

平均：O（n^2）

选排

原理：先选在排，和开头或结尾的数据位置对换

先找出最小(最大)，放到有序数组的开头(结尾)

稳定：都是O(n^2)

插入排序

原理：取头数，作为有序数组第一位，然后后面插入从后往前

最佳O（n）

最差O（n^2）

平均O（n^2）

Shell Sort

原理:改进版的插入排序，引入gap元素

gap = length / 2;

O(log n ^2)

归并排序(稳定)

原理：分组排序的算法，感觉主要就是一种思想（分治法）

你把大数组分成小数组，小数组用的其他算法进行排序，最后再合并成一个大数组

最佳O(N)

最差O（NlogN）

快速排序

原理：（递归）

  1. 从数列中挑一个元素作为“基准”（pivot）

  2. 一般的就是把pivot作为temp，维护两个指针

    然后，前面找小于pivot的数，后面找大于pivot的数，进行交换

3.重复递归

最佳：O（nlogn）

最差：O（n^2）

平均：O（nlogn）

堆排序

原理：

  1. 先把无序数组构建成大堆顶（此时序列的最大值就是堆顶的根节点）
  2. 将根节点和末尾元素交换，则末尾元素成最大值
  3. 将剩余的n-1元素重构成一个新的大堆顶，并重复上面的过程

一般升序用大堆顶，降序用小堆顶

synchronized

为什么用？

并发编程中存在线程安全问题 ：

  1. 存在共享数据，多个线程可能共同操作共享数据

synchronized可以保证同一时刻内只有一个线程操作共享线程

锁对象，锁方法，锁代码块

原理

synchronized保证方法或代码块运行时，同一时刻只有一个方法可以进入临界区

java对象实例的结构

主要由三部分组成：

  • 对象头：

    • 数组长度
    • 对象指针
    • Mark Word: hashCode+分代信息+锁信息

  • 对象实际数据：存放对象的实际数据

  • 对齐填充：对象要求大小为8字节的倍数，不足补齐，所以一个空的Object大小8字节

自旋锁

1.6之前，我们称synchronized是重量级的锁，因为在互斥情况下，没有得到锁的线程就会被挂起来（阻塞），挂起和恢复都需要在kernel中完成，比较臃肿

自旋锁就是为了解决这个问题，如果线程没有获取到锁，被阻塞了，会尝试重新获取锁，进行自旋

默认自旋次数10次。（时间长不好因为自旋会消耗CPU资源）

JDK6后引入了自适应自旋锁，自旋的时间不固定了，而是根据上次自旋的时间和锁的拥有者状态来确定。

锁升级

jdk6后，为了减少获取锁和释放锁带来的性能消耗，引入锁升级的概念

无锁—>偏向锁—>轻量级锁—>重量级锁

锁可以升级不可以降级

“轻量级”是针对和传统的OS互斥而言的，它本质上是在没有多线程竞争的情况下，减少重量级锁的性能消耗

轻量级锁加锁过程
  1. 代码进临界区，如果此时锁状态是0‘01（无锁），虚拟机会先在当前线程的栈帧中创建一个锁记录（Lock Record），用于存储Mark Word的拷贝
  2. 拷贝对象的对象头的Mark Word到Lock Record中
  3. 拷贝成功，尝试将Mark Word的对象指针指向Lock Record，并将Lock Record的owner指针指向Mark Word。成功则4，否则自旋尝试
  4. 成功，那么该线程获得该对象的锁，且Mark word设置为00
  5. 执行完代码块后，释放锁，CAS替换Mark Word的值，如果有线程自旋竞争，CAS肯定失败（Mark Word不会指向当前栈帧，这时候就知道还有竞争，所以就释放锁，并唤醒挂起的线程。

偏向锁

引入偏向锁是为了在无多线程竞争的情况下尽量减少不必要的轻量级锁执行路径

因为轻量级锁会多次CAS，但是偏向锁只会一次CAS

偏向锁获取过程

  1. 访问对象Mark Word中锁信息是否为101，是就不用偏了，如果是001就可以偏向
  2. 判断Mark Word是否指向当前线程ID，如果是，那就是同样的线程进同一个共享资源，自家人进自家门，设成偏向锁101，执行同步代码块
  3. 如果线程ID没有指向当前对象，那么就CAS尝试，竞争成功，则将Mark word指向当前线程ID，执行同步代码块（锁信息101）
  4. 如果竞争失败，则表示有竞争力，当到同步安全点时，获得偏向锁的线程会被挂起，偏向锁在这里升级成轻量级锁。

三种锁的区别

synchronized还是Lock

  1. synchronized是关键字，JVM层面的，内部细节都帮我们做了

Lock是API接口，JDK层面的。

  1. synchronized可以自动释放锁，Lock必须手动释放锁

  2. synchronized不可中断，Lock可中断也可不中断

  3. 通过Lock可以知道线程拿没拿到锁，synchronized不能知道

  4. synchronized可以锁住方法和代码块和对象，Lock只能锁代码块

  5. Lock可以通过读锁提高多线程读的效率

  6. synchronized是非公平锁，Lock可以调制

    其实最大的区别就是 业务是否需要丰富的API.

比如我现在是滴滴，我早上有打车高峰，我代码使用了大量的synchronized，有什么问题？锁升级过程是不可逆的，过了高峰我们还是重量级的锁，那效率是不是大打折扣了？这个时候你用Lock是不是很好？

脱离了业务场景说性能都是耍流氓

ThreadLocal

是什么

ThreadLocal为解决多线程并发提供了另一种思路

传统的多线程同步方式，只有一份共享变量，多个线程排队等待使用该变量，是一种时间换空间的方式，

​ 所以在使用同步机制的时候，要缜密的分析什么时候对变量进行读写，加锁解锁这些方面

ThreadLocal维护变量时，它会为每个变量创建一个线程本地的变量副本，所以每一个线程都可以独立改变自己的副本，就不用考虑同步性的问题，是一种空间换时间的方式，

​ 在用ThreadLocal时，它提供了线程安全的共享对象，所以可以把不安全的变量封装到ThreadLocal中

常用方法（4个）

  • set方法
  • get方法
  • remove方法

remove()：将当前线程的局部变量的值删除，为了减小内存的占用

  • initialValue方法

返回该线程局部变量的初始值

为什么？（原理）

ThreadLocal类提供set/get方法存储和获取value值，但是它本身是不存储值的，ThreadLocal内部维护了一个ThreadLocalMap类，这个类才是真正存储数据的。（key是ThreadLocal实例对象，value是Object对象）

ThreadLocal相关类总结

  • 每个线程是一个Thread实例，其内部维护一个threadLocals的实例成员，其类型是ThreadLocal.ThreadLocalMap。
  • 通过实例化ThreadLocal实例，我们可以对当前运行的线程设置一些线程私有的变量，通过调用ThreadLocal的set和get方法存取。
  • ThreadLocal本身并不是一个容器，我们存取的value实际上存储在ThreadLocalMap中，ThreadLocal只是作为TheadLocalMap的key。
  • 每个线程实例都对应一个TheadLocalMap实例，我们可以在同一个线程里实例化很多个ThreadLocal来存储很多种类型的值，这些ThreadLocal实例分别作为key，对应各自的value，最终存储在Entry table数组中。
  • 当调用ThreadLocal的set/get进行赋值/取值操作时，首先获取当前线程的ThreadLocalMap实例，然后就像操作一个普通的map一样，进行put和get。

Tips: Object和T的区别？

Object是根类，是真实存在的，T只是占位符，仅仅在编译器有效，最终解析时会被擦除，用Object代替

内存泄漏问题

什么是内存泄漏

简单的说，就是东西放在内存里面，但你忘记它放哪里了，它占着一块内存，但是不能回收。当这样的东西越来越多，内存就吃紧，最终导致服务器宕机。

举个例子：抗日上下线

在抗日时期，有两名地下党A和B，A是上线，B是下线，B不能直接联系党中央的，他需要通过A来帮忙传话。一旦A发生意外，党中央就找不到B了，B一直存在，但是茫茫人海，党中央是无法启用B做战斗任务的安排，这种情况类似内存泄漏。

ThreadLocal内存分配
为什么不用强引用？

（1）key 使用强引用

引用ThreadLocal的对象被回收了（栈中的引用结束了）

但是ThreadLocalMap还持有ThreadLocal的强引用，如果没有手动删除，ThreadLocal不会被回收，导致Entry内存泄漏。

（2）key 使用弱引

引用ThreadLocal的对象被回收了，由于ThreadLocalMap持有ThreadLocal的弱引用，即使没有手动删除，ThreadLocal也会被回收。value在下一次ThreadLocalMap调用set、get、remove的时候会被清除。

比较两种情况，我们可以发现：由于ThreadLocalMap的生命周期跟Thread一样长，如果都没有手动删除对应key，都会导致内存泄漏，但是使用弱引用可以多一层保障：弱引用ThreadLocal被清理后key为null，对应的value在下一次ThreadLocalMap调用set、get、remove的时候可能会被清除。

因此，ThreadLocal内存泄漏的根源是：由于ThreadLocalMap的生命周期跟Thread一样长，如果没有手动删除对应key就会导致内存泄漏，而不是因为弱引用。

如何避免内存泄漏

每次使用完ThreadLocal，调用它的remove()方法，清除数据。

ThreadLocal使用场景

  • 当需要存储线程私有变量的时候。
  • 当需要实现线程安全的变量时。
  • 当需要减少线程资源竞争的时候。

强软弱虚

强引用

默认对象都是强引用

String name = “cherrywang”;

User cherrywang = new User();

可达性分析结果可达，不会被GC回收（即使OOM）

可达性分析结果不可达，则GC时会回收

软引用

JVM内存充足，软引用不会被垃圾回收器回收，

当JVM内存不足，才会被垃圾回收器回收

所以软引用经常用来实现一些内存敏感的缓存。只要内存充足就不会被GC掉。

弱引用

弱引用是比软引用生命周期更短的引用。

不论当前内存是否充足，都只能存活到下一次垃圾收集之前。

虚引用

最弱的引用

这种引用不会对对象的生命周期有影响，就跟没有一样

虚引用必须和引用队列一起使用

在对象的垃圾回收过程中，如果GC发现一个对象还存在虚引用，则会把这个虚引用加入到与之关联的引用队列中。

虚引用主要用来跟踪对象被垃圾回收器回收的活动。

它有机会让跟踪的对象“复活”

1. 操作系统概念

OS其实就是管理硬件和软件的程序，它屏蔽了直接操作硬件（直接操作硬件烦的要死）

Kernel

内核决定程序在什么时候对硬件操作多久

内核其实就是完成了对硬件的抽象，给一些硬件API让你调

这样子，可以通过进程间通信和系统调用，进程可间接控制硬件资源。

  1. 内核是OS的核心部分，负责系统的内存管理，硬件设备的管理，文件系统的管理以及应用程序的管理。

  2. 内核连接硬件和程序的桥梁，决定OS的性能和稳定性。

CPU

  1. cpu是运算核心（core）+控制核心（Control Unit），可以称之为计算机的大脑
  2. CPU主要包括控制器+运算器
  3. CPU根本任务是执行指令，一串串0，1序列

CPU与Kernel

  1. Kernel属于OS层面，CPU属于硬件层面
  2. CPU提供运算，处理各种指令，Kernel屏蔽了对硬件的操作，提供四种管理。

2. 同步异步，阻塞非阻塞

2.1 同步异步

同步和异步关注的是消息通信机制 (synchronous communication/ asynchronous communication)

同步：就是在发出一个调用时，在没有得到结果之前，该调用就不返回。（一直等直到要到值为止）
换句话说，就是由调用者主动等待这个调用的结果。(等到它返回值为止)

异步：调用在发出之后，这个调用就直接返回了，所以没有返回结果。

换句话说，当一个异步过程调用发出后，调用者不会立刻得到结果。而是在调用发出后，被调用者通过状态、通知来通知调用者，或通过回调函数处理这个调用。

区别：调用者需不需要等待处理结果，被调用者是否会有回调等机制通知调用者返回结果（针对于被调用者）

举例：打电话问书店老板有没有《操作系统》这本书，如果是同步通信机制，老板会说你等等，我查一下

如果是异步通信机制，老板会说你等等，我查一下等会告诉你，然后把电话挂了

2.2 阻塞非阻塞

阻塞和非阻塞关注的是程序在等待调用结果（消息，返回值）时的状态.

阻塞：阻塞就是发起一个请求，调用者一直等待请求结果返回，当前线程挂起

非阻塞：非阻塞就是发起一个请求，被调用者不用一直等待结果返回。

你打电话问书店老板有没有《分布式系统》这本书，你如果是阻塞式调用，你会一直把自己“挂起”，直到得到这本书有没有的结果，如果是非阻塞式调用，你不管老板有没有告诉你，你自己先一边去玩了， 当然你也要偶尔过几分钟check一下老板有没有返回结果。

3. BIO,AIO,NIO

先了解一下套接字的概念：

套接字（socket）：操作系统为应用程序与TCP／IP协议交互提供了称为套接字 (Socket)的接口，区分不同应用程序进程间的网络通信和连接。

Socket(套接字)可以看成是两个网络应用程序进行通信时，各自通信连接中的端点。

socket 的表现方式 IP:端口

BIO

同步阻塞的IO模型

一连接一线程，有连接请求就要启动一个线程来处理，

​ 客户端发送请求后，先咨询服务端是否有线程相应，如果没有则会一直等待或者遭到拒绝请求

​ 如果有的话，客户端会线程会等待请求结束后才继续执行。

线程不必要开销问题可以通过线程池改善

select的好处是它可以同时监听多个句柄，提高并发量

NIO

同步非阻塞IO模型

一个请求一个线程

​ NIO就是，当一个连接创建后，不需要对应一个线程，该连接会被注册到多路复用器上，所以的连接只需要一个线程就可以搞定

​ 当这个线程中的多路复用器进行轮询的时候，发现连接上有请求的话，才开启一个线程进行处理，也就是一个请求一个线程模式。

​ 适合连接数多且连接短。

AIO

异步非阻塞IO模型

一个有效请求一个线程

客户端的I/O请求都是由OS先完成了再通知服务器应用去启动线程进行处理

用户进程只需要发起一个IO操作然后立即返回，等IO操作真正的完成以后，应用程序会得到IO操作完成的通知，此时用户进程只需要对数据进行处理就好了，程序没有进行实际的读写

三种IO的使用场景

  • BIO方式适用于连接数目比较小且固定的架构，这种方式对服务器资源要求比较高，并发局限于应用中，JDK1.4以前的唯一选择，但程序直观简单易理解。
  • NIO方式适用于连接数目多且连接比较短（轻操作）的架构，比如聊天服务器，并发局限于应用中，编程比较复杂，JDK1.4开始支持。
  • AIO方式使用于连接数目多且连接比较长（重操作）的架构，比如相册服务器，充分调用OS参与并发操作，编程比较复杂，JDK7开始支持。

4. 操作系统基本特性

并发性

  • 并发图

多个事件在同一时间间隔内发生（宏观上是并发的，微观串行）

  • 并行图

多个事件在同一时刻发生

并行需要硬件支持，多核驱动啊什么的CPU

共享性

系统资源可供系统多个并发进程共同使用（资源共享）

  • 互斥共享方式（互斥锁，独占资源，比如说打印机）
  • 同时访问方式（磁盘）

虚拟性

虚拟技术把一个物理实体转换为多个逻辑实体

两种技术：时分复用、空分复用

空分复用：虚拟技术把物理内存抽象为地址空间，进程拥有各自的地址空间，地址空间的页被映射到物理内存中，当用到一个没有在物理内存的页，会执行页面置换算法，将该也置换到内存中。

时分复用：时间片轮转法

异步性

在多道程序环境下允许多个进程并发执行，但只有进程在获得所需的资源后方能执行。

5. 用户态系统态

根据进程访问资源的特点，可分为两个级别：

  • 用户态：用户态运行的程序可以直接读取用户程序的数据
  • 系统态：几乎可以访问计算机的任何资源

一般程序都是在用户态，如果要调用系统资源就需要系统调用

  • 为什么划分两态？

因为有些指令是十分危险的，如果使用不当，可能造成程序崩溃

用户态切换到内存态的方式

  1. 系统调用

系统调用是OS为应用程序提供能够访问到内核态的资源接口

  1. 异常

CPU执行程序时，发生了某些不可知的异常。于是程序进入到处理该异常的内核程序中

  1. 硬件设备的中断

硬件完成用户请求后会发送中断给CPU，CPU会暂停执行下一条指令，转而执行与中断信号对应的应用程序。转换过程涉及到用户态到内核态的转换

6. 程序进程协程

进程

是系统资源分配和调度的基本单位

是程序的一次执行过程，是系统运行程序的基本单位

系统运行一个程序就是一个进程从创建，运行到消亡的过程

程序段，相关的数据段和PCB三部分便构成了进程实体。

PCB:进程控制块，一般在内存中，所谓的进程的创建和销毁都是对PCB的操作。（动态特征的集中反映）

线程

线程是比进程更小的一个单位，是独立调度和分派的基本单位

一个进程在其执行过程可以产生多个线程

引入线程的原因是为了减少程序在并发执行过程中的时空开销

网络socket

一般在TCP/IP通信中比较常见，是OS提供给程序用来与TCP/IP协议交互的接口，socket套接字，两个程序进行网络通信时，通信连接的端点就可以理解为两个socket

socket表现形式 IP：端口

管道

管道分为三种，传输数据都是无类型的，根据FIFO机制

  • 匿名管道：也叫普通管道：PIPE：两个限制：1.只能父子进程，兄弟进程通信 2. 半双工通信，数据只能单向传输（对讲机）
  • 命令流管道：进行了优化，它是全双工通信（电话），但是第一个限制没有解除
  • 命名管道：FIFO，两个限制都解除了

一般就是喜欢拿匿名管道和命名管道做比较

  1. 匿名管道是在内存中的（fork），但是命名管道是在文件系统中的，它是关联了一个路径，以FIFO文件形式存储在文件系统中的，设备文件，所以它打开要用open()才能打开，而且可能阻塞
  2. 匿名管道用pipe函数创建打开，命名函数用mkfifo函数创建，open打开

管道也是文件嘛，大小4096字节，管道满则写阻塞，管道空则读阻塞

系统IPC

消息队列

消息队列遵循FIFO原则

一个系统最多3636个消息队列

一个消息队列所有消息最多可以存放16384个字节

和管道区别：1. 数据有无类型，管道无类型，所以FIFO机制，消息队列有类型，所以可以设优先级机制

  1. 读数据的顺序，管道先来先读，消息队列有优先级机制

信号量

临界资源：同一时刻，只能被同一进程访问的共享资源

临界区：访问临界资源的代码

信号量其实就是一个计数器：记录临界资源能够被多少进程同时访问

一般是用在进程通信方面，或者说同一进程不同线程的同步方法

shmget（）开辟共享内存

shmctl() 删除共享内存

没有连接就会删除，删除后就不会有连接（连接不上）

共享内存是最快的IPC方式，它其实就是映射一段能被其他进程访问的内存。

这个内存是由一个进程创建的，但是其他进程也可以访问。

之所以快是因为不需要客户进程和服务器进程之间进行赋值

共享内存唯一要特别注意的就是同步问题了，所以在这里使用信号量进行共享内存的进程同步是非常常见的。

select poll epoll

IO多路复用的本质就是通过一种机制（系统内核缓存I/O数据），让一个线程能够监视多个文件描述符，一旦某个文件描述符就绪（读或写就绪），能够通知程序进行相应的读写操作。

select，poll，epoll本质上都是同步IO，因为他们都需要在读写事件就绪后自己负责进行读写，也就是说这个读写过程是阻塞的。

和多线程、多进程相比，IO多路复用技术最大的优势就是系统开销小，系统不需要创建进程/线程，也不用维护这些线程/进程，开销大大减少了

文件描述符：

是计算机科学中的一个术语，是一个用于表述指向文件的引用的抽象化概念。

形式上是个非负整数，实际上是一个索引值，指向Kernel中为每一个进程所维护的该进程打开文件的记录表。

当程序打开现有文件或新建一个文件，内核会像进程返回一个文件描述符。

句柄：

door handle 是指门把手，通过门把手可以去控制门，但 door handle 并非 door 本身，只是一个中间媒介。又比如 knife handle 是刀柄，通过刀柄可以使用刀。

跟 door handle 类似，我们可以用 file handle 去操作 file, 但 file handle 并非 file 本身。这个 file handle 就被翻译成文件句柄，同理还有各种资源句柄。

maxfdp1：制定待测试的文件描述符个数，值=待测个数+1

中间三个set，理解为一个集合，存放文件描述符，即文件句柄

fd_set *readset 读集合

fd_set *writeset 写集合

fd_set *exceptset 异常集合

const struct timeval *timeout 可等待多少时间

返回int，若有就绪描述符返回个数，超时0，出错-1

select机制

select()的机制提供一种fd_set的数据结构，实际上是一个long类型数组，每一个元素都能和一打开的句柄。

当调用select（）时，由内核根据IO状态修改fd_set的内容，由此来通知执行了select的socket或文件哪个可读

select的缺陷
  1. 每次调用select（），都需要将fd_set集合从用户态拷贝到内核态中，如果fd_set开销很大，那么这个开销也很大
  2. 每次调用select（），需要遍历从内核传递过来的fd_set，如果fd_set很大，开销也大
  3. select（)监视的fd_set大小有限制（1024，宏定义的）
Poll

poll的机制和select本质上没有什么区别，

它将用户传入的数组拷贝到内核空间，然后查询每个fd对应的设备状态，如果设备就绪则在设备等待队列中加入一项并继续遍历，如果遍历完所有fd后没有发现就绪设备，则挂起当前进程，直到设备就绪或者主动超时，被唤醒后它又要再次遍历fd。这个过程经历了多次无谓的遍历。

poll的优化就是没有大小限制

即Poll解决了上述的问题3，但是问题1，问题2的性能开销问题没有解决。

pollfd替换了fd_set，使得文件描述符集合的限制远大于1024(原因是它是基于链表来存储的)

struct pollfd *fds： 用于存放需要检测的socket描述符

nfds_t nfds： 记录数组fds中描述符的总数量

返回fds集合中就绪的读、写，或出错的描述符数量，返回0表示超时，返回-1表示出错；

  1. poll的特点是“水平触发”`，如果报告了fd后，没有被处理，那么下次poll时会再次报告该fd。

注意：从上面看，select和poll都需要在返回后，通过遍历文件描述符来获取已经就绪的socket。事实上，同时连接的大量客户端在一时刻可能只有很少的处于就绪状态，因此随着监视的描述符数量的增长，其效率也会线性下降。

epoll

epoll是poll和select的进化版

相对于select来说，epoll没有描述符个数限制

使用一个文件描述符管理多个描述符，将用户关心的文件描述符的事件存放到内核的一个事件表中，这样在用户空间和内核空间的copy只需一次。

epoll_create： 创建一个epoll句柄，size表明内核监听的描述符数量

调用成功返回一个epoll的句柄描述符，失败-1

epoll_ctl： 注册要监听的事件类型

epfd：表示epoll句柄

op：表示fd操作，三种

  • EPOLL_CTL_ADD：注册新的fd到epfd中
  • EPOLL_CTL_MOD：修改已注册的fd监听事件
  • EPOLL_CTL_DEL：从epfd中删除一个fd

fd：要监听的描述符

event：要监听的事件

epoll_wait：等待事件的就绪，成功返回事件数目，失败-1，超时0

epfd：epoll句柄

events：从内核得到的就绪事件集合

maxevents：告诉内核events的大小

timeout：等待超时时间

epoll能显著提高程序在大量并发连接中只有少量活跃的情况下的系统CPU利用率。原因就是获取事件的时候，它无须遍历整个被侦听的描述符集，只要遍历那些被内核IO事件异步唤醒而加入Ready队列的描述符集合就行了。

epoll有两种触发方式（多了边缘触发，水平触发都有）

水平触发（LT）：默认工作模式，即当epoll_wait检测到某描述符事件就绪并通知应用程序时，应用程序可以不立即处理该事件；下次调用epoll_wait时，会再次通知此事件

边缘触发（ET）： 当epoll_wait检测到某描述符事件就绪并通知应用程序时，应用程序必须立即处理该事件。如果不处理，下次调用epoll_wait时，不会再次通知此事件。（直到你做了某些操作导致该描述符变成未就绪状态了，也就是说边缘触发只在状态由未就绪变为就绪时只通知一次）。

红黑树

特性
  • 每个节点是红色或者黑色（非红即黑）
  • 根节点是黑色（Root黑）
  • 每一个空叶子节点必须是黑色（NULL黑）
  • 红色节点的子节点必须是黑色（暴躁红）
  • 节点中到达任意子节点包含相同数组的黑节点
    • 最长路径的节点总数量不会超过最短路径的两倍

插入

putTreeVal（）:插入节点

简单来说

1、从root节点开始寻找 如果目标 k.hash 小于 当前节点的 hash ,那么到左树寻找，大于那么从右树寻找。如果找到key相同并且equals相同的节点 p 那就直接返回。
2、如果hash相同 但是equal不同 进而通过Comparable接口，进行比较，如果比较的结果如果还是相等，则在左右子树递归的寻找是否有与要插入的key equals相同的元素。如果有那么直接return返回。
（也即是没实现Comparable接口，大小由hash判定。实现了，则由Comparable接口的比较方法判定）
3、如果遍历完所有的节点 并未找到equals相同的节点。那就需要插入该新节点。必须分出大小，所以通过执行tieBreakOrder方法，该方法的返回值是-1，1。如果是-1则插入到左边节点，1就插入到右边节点。
4、插入完成之后，需要重新移动root节点 到table数组的i位置的第一个节点上 并且需重新平衡红黑树。

平衡

balanceInsertion（）方法

左旋右旋

其实每次重新插入节点，就要进行染色，平衡

新节点默认颜色是红色的，因为红色不会影响路径上黑色节点的数量，保持性质5。

然后根据性质进行染色，通过左旋右旋进行平衡

红黑树链化

一般在扩容时才能退化成链表

  1. 扩容时如果是红黑树结构会执行红黑树的 split( ) 方法

  2. split 方法中会初始化生成 loHead 和 hiHead 两个红黑树的头结点（之后会用 low 和 high 表示）
    lc 和 hc 分别为 low 和 high 的元素个数，初始化为0

  3. 把红黑树中的结点依次添加到 low 和 high 两颗红黑树中，依靠 (e.hash & bit) == 0 的位运算来判断属于哪颗树，bit是传过来的旧数组下标。每颗树元素的增加都会使对应的 lc 或 hc 自增1。

  4. 生成完 low 和 high 两颗红黑树后，如果对应的元素个数 lc，hc 小于等于6，退化成链表，否则维持红黑树形态。

  5. low树插入到新数组 tab[index] 的位置上，index是当前红黑树所在旧数组坐标；high树插入到新数组 tab[index + bit] 的位置上，bit是旧数组长度。

https://www.jianshu.com/p/8a58d8335270

jmm是什么？

讲述jmm的背景

原来CPU和内存差不多，后来CPU快了，就增加了工作内存（寄存器和高速缓存）多核并发缓存架构

jmm解决的问题？

为了解决硬件和操作系统的差异

jmm的各部分是干嘛的

分为主内存，工作内存和java线程

主内存是公用的，一般主内存就是我们所说的物理内存，然后我们一般是不对主内存直接修改的

如果每次java线程的更改都直接操作主内存，那么性能影响是比较大的。

为了解决这个问题，工作内存就出现了，工作内存就是我们所说的寄存器和高速缓存。

java线程对变量的操作，会在工作内存中进行操作，不能去直接操作主内存

但是这样子我们能感觉到肯定会有线程不安全的问题。

如何解决线程不安全的问题？

回答这个问题前，我们必须先知道

JMM针对于内存，设定了8个指令

  • lock
  • unlock

作用于主内存，是否将一个变量标识为内存独占

  • read
  • load

作用于工作内存，read就是从主内存get数据，load就相当于将主内存数据set到工作内存中

  • use：使用
  • assign：赋值

作用于工作内存，use把工作内存的变量传给执行引擎，assign从执行引擎接收到的值放入到工作内存的变量副本中

  • store
  • write

作用于主内存，store将工作内存的数据传送到主内存中，write就是将传送到主内存中的数据写入到主内存中

JMM对这八个命令指定了很多规则，但是一般都是按照happen-before规则进行执行的。

JMM的模型特征

原子性

JMM保证上面说的8个交互操作指令都是具有原子性的

但是有例外就是，JMM允许虚拟机把没有volatile修饰的64位数据拆分成两个32位数据来进行操作。

其实就是单个指令是具有原子性的，但是多个指令在整体上是没有原子性的。

解决方案：1. 用原子类 2.synchronzied互斥锁。

原子类内部实现是CAS （自旋锁），乐观锁

synchronized内部实现是monitor锁，是悲观锁

可见性

可见性就是说一个线程修改了值，能够让其他线程知道并修改

JMM实现的途径是变量修改后会将新值同步到主内存中，在变量读取前从主内存中刷新变量的值来实现可见性

实现可见性的方案：

  • volatile关键字

volatile关键字的两个作用，一个是保证变量的可见性，一个是防止指令重排序

保证变量可见性：一个变量，被volatile修饰，就意味着线程的本地内存是无效的，当java线程更改该变量，会直接将新值更新到主内存中；其他线程读取该变量时，也会从主内存中直接读取（因为本地内存没有呀）

  • synchronzied

unlock操作之前，必须把变量值同步到主内存中

  • final关键字

final关键字初始化完成，且没有发生this逃逸，其他线程就能看见final修饰的值

有序性

在单线程环境下，程序永远会“有序的”执行，即：线程内表现为串行语义。但是在多线程环境下，程序会出现“乱序”。

这是因为，JMM允许编译器和处理器对指令进行重排序，本意是为了提高效率，但是在多线程并发的情况下，有可能影响并发执行的正确性

解决方案：

  • volatile关键字

volatie的另一个作用，禁止指令重排序，它是通过添加内存屏障的方式来禁止指令重拍序的。

重排序时，不能把后面的指令放到内存屏障前。

  • synchronized

它通过加锁，让一段代码只能由一个线程来执行。

这两个区别就是，synchronized可以修饰一段代码，或者一个方法。但是volatile只能修饰一个变量。

线程通信方式

volatile关键字

该关键字有两个特性，一：可见性；二：禁止指令重排序

其中可见性就是让线程之间进行通信，两个原则保证线程可见性

  • 所有volatile修饰的变量一旦被某个线程更改，必须立即刷新到主内存
  • 所有volatile修饰的变量在使用之前必须重新读取主内存的值
等待通知机制

等待通知机制是通过wait和notify方法来实现的

wait()：当线程调用wait方法，会释放当前的锁，然后让出CPU，进入等待状态

notify()：唤醒沉睡的线程（不会释放锁）

wait要配合synchronized一起使用，如果没有synchronized会报错

既然wait和notify配合synchronized使用能够实现,当然用 ReentrantLock 结合 Condition也可以完成;

join（）

退让方法

当线程A调用另一个线程B的join方法时，当前线程A阻塞等待线程B执行完毕才能继续执行

相当于A退让了，让B先进行

所以join的好处能够保证线程的执行顺序，但是如果调用线程的join方法其实已经失去了并行的意义

虽然存在多个线程，但是本质上还是串行的

最后join的实现其实是基于等待通知机制的。

两种工具类

JUC工具类CountDownLatch

使用一个计数器进行实现。计数器初始值为线程的数量。

当每一个线程完成自己任务后，计数器的值就会减一。

当计数器的值为0时，表示所有的线程都已经完成一些任务，然后在CountDownLatch上等待的线程就可以恢复执行接下来的任务。

CountDownLatch典型用法：1、某一线程在开始运行前等待n个线程执行完毕。将CountDownLatch的计数器初始化为new CountDownLatch(n)，每当一个任务线程执行完毕，就将计数器减1 countdownLatch.countDown()，当计数器的值变为0时，在CountDownLatch上await()的线程就会被唤醒。一个典型应用场景就是启动一个服务时，主线程需要等待多个组件加载完毕，之后再继续执行。

LockSupport

它是一个线程工具类，所有的方法都是静态方法，可以让线程在任意位置阻塞，也可以在任意位置唤醒。

它的内部其实两类主要的方法：park（停车阻塞线程）和unpark（启动唤醒线程）。

（1）wait和notify都是Object中的方法,在调用这两个方法前必须先获得锁对象，但是park不需要获取某个对象的锁就可以锁住线程。

（2）notify只能随机选择一个线程唤醒，无法唤醒指定的线程，unpark却可以唤醒一个指定的线程。
