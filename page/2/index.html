JMM

https://www.jianshu.com/p/8a58d8335270

jmm是什么？

讲述jmm的背景

原来CPU和内存差不多，后来CPU快了，就增加了工作内存（寄存器和高速缓存）多核并发缓存架构

jmm解决的问题？

为了解决硬件和操作系统的差异

jmm的各部分是干嘛的

分为主内存，工作内存和java线程

主内存是公用的，一般主内存就是我们所说的物理内存，然后我们一般是不对主内存直接修改的

如果每次java线程的更改都直接操作主内存，那么性能影响是比较大的。

为了解决这个问题，工作内存就出现了，工作内存就是我们所说的寄存器和高速缓存。

java线程对变量的操作，会在工作内存中进行操作，不能去直接操作主内存

但是这样子我们能感觉到肯定会有线程不安全的问题。

如何解决线程不安全的问题？

回答这个问题前，我们必须先知道

JMM针对于内存，设定了8个指令

  • lock
  • unlock

作用于主内存，是否将一个变量标识为内存独占

  • read
  • load

作用于工作内存，read就是从主内存get数据，load就相当于将主内存数据set到工作内存中

  • use：使用
  • assign：赋值

作用于工作内存，use把工作内存的变量传给执行引擎，assign从执行引擎接收到的值放入到工作内存的变量副本中

  • store
  • write

作用于主内存，store将工作内存的数据传送到主内存中，write就是将传送到主内存中的数据写入到主内存中

JMM对这八个命令指定了很多规则，但是一般都是按照happen-before规则进行执行的。

JMM的模型特征

原子性

JMM保证上面说的8个交互操作指令都是具有原子性的

但是有例外就是，JMM允许虚拟机把没有volatile修饰的64位数据拆分成两个32位数据来进行操作。

其实就是单个指令是具有原子性的，但是多个指令在整体上是没有原子性的。

解决方案：1. 用原子类 2.synchronzied互斥锁。

原子类内部实现是CAS （自旋锁），乐观锁

synchronized内部实现是monitor锁，是悲观锁

可见性

可见性就是说一个线程修改了值，能够让其他线程知道并修改

JMM实现的途径是变量修改后会将新值同步到主内存中，在变量读取前从主内存中刷新变量的值来实现可见性

实现可见性的方案：

  • volatile关键字

volatile关键字的两个作用，一个是保证变量的可见性，一个是防止指令重排序

保证变量可见性：一个变量，被volatile修饰，就意味着线程的本地内存是无效的，当java线程更改该变量，会直接将新值更新到主内存中；其他线程读取该变量时，也会从主内存中直接读取（因为本地内存没有呀）

  • synchronzied

unlock操作之前，必须把变量值同步到主内存中

  • final关键字

final关键字初始化完成，且没有发生this逃逸，其他线程就能看见final修饰的值

有序性

在单线程环境下，程序永远会“有序的”执行，即：线程内表现为串行语义。但是在多线程环境下，程序会出现“乱序”。

这是因为，JMM允许编译器和处理器对指令进行重排序，本意是为了提高效率，但是在多线程并发的情况下，有可能影响并发执行的正确性

解决方案：

  • volatile关键字

volatie的另一个作用，禁止指令重排序，它是通过添加内存屏障的方式来禁止指令重拍序的。

重排序时，不能把后面的指令放到内存屏障前。

  • synchronized

它通过加锁，让一段代码只能由一个线程来执行。

这两个区别就是，synchronized可以修饰一段代码，或者一个方法。但是volatile只能修饰一个变量。

线程通信方式

volatile关键字

该关键字有两个特性，一：可见性；二：禁止指令重排序

其中可见性就是让线程之间进行通信，两个原则保证线程可见性

  • 所有volatile修饰的变量一旦被某个线程更改，必须立即刷新到主内存
  • 所有volatile修饰的变量在使用之前必须重新读取主内存的值
等待通知机制

等待通知机制是通过wait和notify方法来实现的

wait()：当线程调用wait方法，会释放当前的锁，然后让出CPU，进入等待状态

notify()：唤醒沉睡的线程（不会释放锁）

wait要配合synchronized一起使用，如果没有synchronized会报错

既然wait和notify配合synchronized使用能够实现,当然用 ReentrantLock 结合 Condition也可以完成;

join（）

退让方法

当线程A调用另一个线程B的join方法时，当前线程A阻塞等待线程B执行完毕才能继续执行

相当于A退让了，让B先进行

所以join的好处能够保证线程的执行顺序，但是如果调用线程的join方法其实已经失去了并行的意义

虽然存在多个线程，但是本质上还是串行的

最后join的实现其实是基于等待通知机制的。

两种工具类

JUC工具类CountDownLatch

使用一个计数器进行实现。计数器初始值为线程的数量。

当每一个线程完成自己任务后，计数器的值就会减一。

当计数器的值为0时，表示所有的线程都已经完成一些任务，然后在CountDownLatch上等待的线程就可以恢复执行接下来的任务。

CountDownLatch典型用法：1、某一线程在开始运行前等待n个线程执行完毕。将CountDownLatch的计数器初始化为new CountDownLatch(n)，每当一个任务线程执行完毕，就将计数器减1 countdownLatch.countDown()，当计数器的值变为0时，在CountDownLatch上await()的线程就会被唤醒。一个典型应用场景就是启动一个服务时，主线程需要等待多个组件加载完毕，之后再继续执行。

LockSupport

它是一个线程工具类，所有的方法都是静态方法，可以让线程在任意位置阻塞，也可以在任意位置唤醒。

它的内部其实两类主要的方法：park（停车阻塞线程）和unpark（启动唤醒线程）。

（1）wait和notify都是Object中的方法,在调用这两个方法前必须先获得锁对象，但是park不需要获取某个对象的锁就可以锁住线程。

（2）notify只能随机选择一个线程唤醒，无法唤醒指定的线程，unpark却可以唤醒一个指定的线程。

AQS

AbstractQueuedSynchronizer 队列同步器

是一个用来构建锁和同步器的框架

AQS可以构建大量广泛的同步器

常见：

ReentrantLock

CyclicBarrier

Semaphore

FutureTask

CountDownLatch

Condition

原理

如果共享资源空闲，有线程试图获取该资源，就会将该线程设为有效的工作线程。

如果共享资源被占用，那么就需要一套线程阻塞等待以及唤醒时锁分配机制，该机制是AQS中的CLH队列锁（双端虚拟队列）实现的。

CLH：一个虚拟双向队列（不存在队列实例，仅有节点相连）

AQS将每条请求共享资源的线程封装成CLH中的Node节点来实现锁的分配

AQS通过int成员变量表示同步状态，内置FIFO队列完成资源线程的排队工作。

AQS使用CAS对同步状态进行原子操作实现state的更改

两种共享方式

Exclusive（独占）

只有一个线程能执行，如ReentrantLock。可分为公平锁和非公平锁

公平锁：先到先得锁

非公平锁：无视顺序直接抢锁，谁先抢到就是谁的

ThreadPoolExecutor和ThreadPoolTaskExecutor的区别

有一个需求，需要对日志进行批量的插入到数据库中。数据量多的话，更倾向的选择就是用异步的方式调用DAO方法，批量多线程插入。

这里就碰到一个问题，使用哪个线程池，本来就只认识ThreadPoolExecutor这个线程池，所以去百度搜有没有相关的代码学习一下。

没想到有很多的代码是关于ThreadPoolTaskExecutor这个线程池的代码

所以就简单看看

首先是ThreadPoolExecutor线程池

这个线程池是JDK中的线程池类，继承自Executor接口（该接口类似于一个线程池工厂类），里面有execute()方法用来执行线程，线程池会提供一个任务队列，队列中是正在排队的等待线程。

执行流程加个配图

然后是ThreadPoolTaskExecutor线程池

这个线程池是Spring包下的线程池，是Spring为我们提供的线程池类

我们可以通过配置类进行线程池的注入

区别：

1、ThreadPoolTaskExecutor使用ThreadPoolExecutor并增强，扩展了更多特性

2、ThreadPoolTaskExecutor只关注自己增强的部分，任务执行还是ThreadPoolExecutor处理。

3、前者spring自己用着爽，后者离开spring我们用ThreadPoolExecutor爽。

注意：ThreadPoolTaskExecutor 不会自动创建ThreadPoolExecutor需要手动调initialize才会创建

如果@Bean 就不需手动，会自动InitializingBean的afterPropertiesSet来调initialize

这是写的测试类，不知道为什么用线程池调用的话会爆InterruptException。。

大师我悟了！！

报InterruptException的原因是因为单元测试的原因我们的Runnable线程在Sleep的过程中，单元测试其实就跑完了，跑完了它会关闭Tomcat服务器，线程池在察觉到这个单元测试关闭的请求后，会强制线程中断，从而造成InterruptException的异常问题。

解决方案，在线程外再睡一会儿，给足够的时间让线程sleep。

SYN攻击

有时会被问到了不了解SYN攻击

写之前，其实并不了解，希望写完之后能够留下一点印象

概念

​ SYN泛洪攻击，利用的是TCP的三次握手机制

​ 攻击端利用伪造的IP地址向被攻击端发出请求，服务端的响应请求永远都到不了目的地，服务端会在等待关闭连接的过程中损耗了资源。

​ 如果请求成千上万的话，主机资源将会被耗尽，从而达到攻击的目的

正常的TCP三次握手

DOS攻击

攻击方不执行第三次握手，使得server端处于半连接状态

解决措施

①减少SYN重试次数/降低timeout时间

②增大backlog队列（TCP中有两个backlog队列存放半连接队列和全连接队列）

其中半连接队列的backlog默认值为2048，全连接队列默认值为128

③超出处理能力，直接丢弃新来的SYN

④SYN cookie(等建立起连接在分配数据区，依赖对方使用真实的IP，不然这个方法也不顶用。。)

​ 原本tcp协议是在收到syn包时，服务器返回syn+ack包并分配一个专门的数据区来储存tcp连接需要的数据，这就使攻击者有机可乘，可以利用伪造的syn包来消耗服务器的内存空间和半开连接数，这就可以使服务器的内存或者半开连接数耗尽而拒绝服务。

​ 而syn cookie技术是服务器在收到syn包时并不马上分配储存连接的数据区，而是根据这个syn包计算出一个cookie，把这个cookie填入tcp的Sequence Number字段发送syn+ack包，等对方回应ack包时检查回复的Acknowledgment Number字段的合法性，如果合法再分配专门的数据区。

Socket中close（）和shutdown（）的区别

编写并发服务器最简单的方法就是fork一个子进程来服务每个客户。

当一个连接建立时，accept返回，服务器接着调用fork创建子进程通过connfd套接字服务客户端

父进程继续等待另一个连接（通过监听套接字listenfd）。

新客户由子进程提供服务，父进程就会关闭自己和服务的socket

举个栗子：

  1. 服务器收到一个客户端请求
  1. 服务器调用fork，此时子进程会复制父进程的全部信息，包括套接字

  1. 父进程关闭连接套接字（connfd），子进程关闭监听套接字（listenfd)

最终就是子进程来处理这个新进来的服务，父进程可以在监听套接字上调用accept来处理下一个客户连接

close函数

close函数是用来关闭套接字的

成功返回0，失败返回-1

TCP如果说要停止通信，会发送一个FIN命令嘛，但是什么时候发送FIN命令，是看引用计数是否为0,

close()只是导致相应描述符的引用计数-1，但是如果引用计数扔>0，还是不会发FIN

如果想要强制发送FIN，可以使用shutdown()函数。

如上图，在第二步中，connfd和listenfd都为2

第三步，父进程关闭connfd，子进程关闭listenfd，这样子connfd和listenfd都为1

而该套接字真正的关闭是要引用计数==0才关闭。

即close只要一到0就关闭，是粗暴式的关闭。

shutdown函数

成功返回0，失败返回-1

该函数的行为依赖howto参数的值

  • SHUT_RD（读关闭）

套接字中不再有数据可接收，对该套接字进行read直接返回EOF

而且套接字接收缓冲区中的现有数据都被丢弃。

  • SHUT_WR（写关闭）

对于TCP套接字，称为半关闭（half-close）。

当前在套接字发送缓冲区中的数据将被发送掉，后跟TCP的正常连接终止序列。（不管套接字的引用计数是否等于0）

  • SHUT_RDWR

等于调用shutdown函数两次，连接的读半部和写半部都关闭。

shutdown可以不管引用计数就激发TCP的正常连接终止序列，比较优雅

  • shutdown是一种优雅地单方向或者双方向关闭socket的方法。 而close则立即双方向强制关闭socket并释放相关资源。

  • 如果有多个进程共享一个socket，shutdown影响所有进程，而close只影响本进程。

  • 其实就是半双工和全双工的区别

IOC

IOC 控制反转。它是一种思想而不是一个技术实现，描述的是：Java开发领域对象的创以及管理问题

控制反转：控制对象实例化，管理的能力反转交给IOC容器处理

举个栗子：当前A依赖于B

传统思想，我们会在A中手动new一个B对象出来

IOC：不用new关键字创建对象，而是通过IOC容器帮助我们实例化对象。

IOC解决问题

IOC的思想就是由第三方容器来管理相关资源，使得类之间耦合度降低

好处：

  1. 对象之间的耦合度降低
  2. 资源变得容易管理，比如Spring容器自动装配，自动就实现了一个单例模式（@Bean的都是单例）

举个栗子：现在有一个User，通过Service和Dao两层进行开发

如果没有IOC思想的情况下，需要在UserServiceImpl中手动new出Dao层的实现类。

咋一看，发现和自动装配也没什么差，但是思考一个场景

现在有个新需求，对IUserDao层开发出新的实现类。我们Service层依赖的是具体的实现，当需求需要改变，需要修改具体的实现类，如果有很多地方引用了IUserDao的具体实现的话，修改起来非常麻烦

如果使用IOC

@Autowired

private IUserDao userDao;

  • 看到一个很合适的说法

1、IoC(控制反转)
　　首先想说说IoC（Inversion of Control，控制反转）。这是spring的核心，贯穿始终。所谓IoC，对于spring框架来说，就是由spring来负责控制对象的生命周期和对象间的关系。这是什么意思呢，举个简单的例子，我们是如何找女朋友的？常见的情况是，我们到处去看看找到自己喜欢的，然后打听她们的兴趣爱好、qq号、电话号………，想办法认识她们，投其所好送其所要……这个过程是复杂深奥的，我们必须自己设计和面对每个环节。传统的程序开发也是如此，在一个对象中，如果要使用另外的对象，就必须得到它（自己new一个），使用完之后还要将对象销毁，对象始终会和其他的接口或类耦合起来。

那么IoC是如何做的呢？有点像通过婚介找女朋友，在我和女朋友之间引入了一个第三者：婚姻介绍所。婚介管理了很多男男女女的资料，我可以向婚介提出一个要求的列表，告诉它我想找个什么样的女朋友，然后婚介就会按照我们的要求，提供一个女孩，我们只需要去和她谈恋爱、结婚就行了。简单明了，如果婚介给我们的人选不符合要求，我们就会抛出异常。整个过程不再由我自己控制，而是有婚介这样一个类似容器的机构来控制。Spring所倡导的开发方式就是如此，所有的类都会在spring容器中登记，告诉spring你是个什么东西，你需要什么东西，然后spring会在系统运行到适当的时候，把你要的东西主动给你，同时也把你交给其他需要你的东西。所有的类的创建、销毁都由 spring来控制，也就是说控制对象生存周期的不再是引用它的对象，而是spring。对于某个具体的对象而言，以前是它控制其他对象，现在是所有对象都被spring控制，所以这叫控制反转。

2、DI(依赖注入)
　　IoC的一个重点是在系统运行中，动态的向某个对象提供它所需要的其他对象。这一点是通过DI（Dependency Injection，依赖注入）来实现的。比如对象A需要操作数据库，以前我们总是要在A中自己编写代码来获得一个Connection对象，有了 spring我们就只需要告诉spring，A中需要一个Connection，至于这个Connection怎么构造，何时构造，A不需要知道。在系统运行时，spring会在适当的时候制造一个Connection，然后像打针一样，注射到A当中，这样就完成了对各个对象之间关系的控制。A需要依赖 Connection才能正常运行，而这个Connection是由spring注入到A中的，依赖注入的名字就这么来的。那么DI是如何实现的呢？ Java 1.3之后一个重要特征是反射（reflection），它允许程序在运行的时候动态的生成对象、执行对象的方法、改变对象的属性，spring就是通过反射来实现注入的。

OOP思想可以解决大部分代码重复问题。但是也会有处理不了的情况，比如：横切代码逻辑

横切逻辑代码的问题：

  • 代码重复问题
  • 横切逻辑代码和业务逻辑混杂在一起，代码臃肿，不利于维护

AOP 主要用来解决：在不改变原有业务逻辑的情况下，增强横切逻辑代码，根本上解耦合，避免横切逻辑代码重复。
